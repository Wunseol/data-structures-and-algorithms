# 06 - 哈希表模块需求文档

## 1. 模块概述

哈希表（Hash Table）是计算机科学中最重要的数据结构之一，提供了近乎 O(1) 的查找、插入和删除性能。在现代软件系统中，哈希表无处不在：

- **编程语言内置**：C++ `std::unordered_map`/`std::unordered_set`、Python `dict`/`set`、Java `HashMap`/`HashSet`、Go `map`
- **数据库系统**：MySQL InnoDB 的自适应哈希索引、Redis 的键值存储
- **缓存系统**：Memcached、LRU Cache
- **网络系统**：DNS 缓存、路由表查找
- **安全领域**：密码哈希、布隆过滤器（垃圾邮件过滤、URL 去重）
- **分布式系统**：一致性哈希（负载均衡、分布式缓存）

本模块为全新模块，项目当前无任何哈希表相关代码。需从零构建完整的哈希表知识体系，从基础哈希函数设计到高级应用（布隆过滤器、一致性哈希），以 C/C++ 为核心实现语言，Python 为辅助对比验证语言。

## 3. 新增需求

### 3.1 哈希函数设计

#### 3.1.1 直接定址法

**需求描述**：使用关键字本身或线性函数作为哈希地址。

**实现要求**：

- 哈希函数：`H(key) = key` 或 `H(key) = a * key + b`
- 适用场景：关键字分布连续且范围不大
- 优缺点分析

**验收标准**：
- 无冲突（关键字不重复时）
- 空间利用率低时能正确分析

#### 3.1.2 除留余数法

**需求描述**：取关键字被某个不大于表长的质数除后的余数作为哈希地址。

**实现要求**：

- 哈希函数：`H(key) = key % p`，其中 `p` 为不大于表长 `m` 的最大质数
- 质数选择策略：预计算常用表长对应的最佳质数
- 支持整数类型关键字
- 分析不同 `p` 值对冲突分布的影响

**验收标准**：
- 哈希值在 `[0, m-1]` 范围内
- 使用质数作为模数时冲突率低于使用合数
- 提供冲突分布统计

#### 3.1.3 乘法哈希

**需求描述**：使用关键字乘以一个常数后取小数部分再乘以表长。

**实现要求**：

- 哈希函数：`H(key) = floor(m * (key * A - floor(key * A)))`，其中 `A` 为黄金比例相关常数 `(√5 - 1) / 2 ≈ 0.618`
- 分析乘法哈希对关键字模式的鲁棒性
- 与除留余数法的冲突率对比

**验收标准**：
- 哈希值在 `[0, m-1]` 范围内
- 对连续关键字序列的冲突分布均匀
- 提供与除留余数法的冲突率对比数据

#### 3.1.4 全域哈希

**需求描述**：从一组哈希函数中随机选择，避免恶意输入导致的性能退化。

**实现要求**：

- 哈希函数族：`H(key) = ((a * key + b) % p) % m`，其中 `p` 为大于所有关键字的质数，`a ∈ [1, p-1]`，`b ∈ [0, p-1]`
- 随机选择 `a` 和 `b`
- 证明：对任意两个不同的关键字，冲突概率 ≤ 1/m

**验收标准**：
- 随机选择哈希函数后冲突概率 ≤ 1/m
- 对抗性测试：构造最坏情况输入，验证全域哈希仍保持 O(1) 期望性能
- 提供概率分析

**技术参考**：
- [CLRS](https://mitpress.mit.edu/9780262046305/introduction-to-algorithms/) 第 11.3 节 — 全域哈希的数学证明
- [cppreference: std::hash](https://en.cppreference.com/w/cpp/utility/hash) — C++ 标准库哈希函数对象

### 3.2 冲突解决

#### 3.2.1 链地址法（Separate Chaining）

**需求描述**：每个桶存储一个链表，冲突元素追加到链表尾部。

**实现要求**：

- 数据结构：
  ```c
  typedef struct HashNode {
      int key;
      int value;
      struct HashNode *next;
  } HashNode;

  typedef struct {
      HashNode **buckets;  // 指针数组
      int size;            // 当前元素数
      int capacity;        // 桶数量
  } HashTable;
  ```
- 基本操作：
  - `create(capacity)`：创建哈希表
  - `insert(key, value)`：插入键值对
  - `search(key)`：查找键
  - `delete(key)`：删除键
  - `destroy()`：销毁哈希表
- 支持动态扩容（见 3.4 节）
- 统计链长分布

**验收标准**：
- 插入/查找/删除操作正确
- 无内存泄漏
- 平均查找长度与理论值一致：ASL ≈ 1 + α/2（成功），ASL ≈ α + e^α（失败），α 为装填因子
- 处理空表、单元素、大量冲突等边界情况

#### 3.2.2 开放地址法（Open Addressing）

**需求描述**：冲突时按探测序列寻找下一个空槽位。

**实现要求**：

- 数据结构：
  ```c
  typedef struct {
      int key;
      int value;
      int is_occupied;   // 0: 空, 1: 已占用, 2: 已删除（墓碑标记）
  } HashEntry;

  typedef struct {
      HashEntry *entries;
      int size;
      int capacity;
  } HashTable;
  ```

- 三种探测方式：

  **a) 线性探测（Linear Probing）**
  - 探测序列：`H(key), H(key)+1, H(key)+2, ...`
  - 优点：缓存友好
  - 缺点：一次聚集（Primary Clustering）
  - ASL：成功 ≈ (1 + 1/(1-α))/2，失败 ≈ (1 + 1/(1-α)²)/2

  **b) 二次探测（Quadratic Probing）**
  - 探测序列：`H(key), H(key)+1², H(key)+2², H(key)+3², ...`
  - 优点：缓解一次聚集
  - 缺点：二次聚集（Secondary Clustering）
  - 要求：表长为 4k+3 形式的质数，且装填因子 α < 0.5

  **c) 双重哈希（Double Hashing）**
  - 探测序列：`(H1(key) + i * H2(key)) % m`，`i = 0, 1, 2, ...`
  - `H2(key) = 1 + (key % (m-1))`（确保 H2 ≠ 0）
  - 优点：探测序列最多，冲突率最低
  - 缺点：两次哈希计算开销

- 墓碑标记（Tombstone）：删除操作使用墓碑标记而非直接清空，保证查找链不断裂
- 墓碑清理策略：定期清理或插入时复用

**验收标准**：
- 三种探测方式均正确实现
- 线性探测出现一次聚集现象（可通过链长分布统计验证）
- 二次探测在 α < 0.5 时保证能找到空槽
- 双重哈希冲突率低于线性探测和二次探测
- 墓碑标记正确处理，查找链不断裂
- 提供三种探测方式的 ASL 对比实验

**技术参考**：
- [CLRS](https://mitpress.mit.edu/9780262046305/introduction-to-algorithms/) 第 11.2-11.4 节
- [cppreference: std::unordered_map](https://en.cppreference.com/w/cpp/container/unordered_map) — C++ 标准库使用链地址法实现

### 3.3 装填因子与性能分析

**需求描述**：系统分析装填因子对哈希表性能的影响。

**实现要求**：

- 装填因子定义：`α = n / m`（n 为元素数，m 为桶数）
- 理论分析：
  - 链地址法：ASL_成功 ≈ 1 + α/2，ASL_失败 ≈ α + e^α
  - 开放地址法（线性探测）：ASL_成功 ≈ (1 + 1/(1-α))/2，ASL_失败 ≈ (1 + 1/(1-α)²)/2
  - 开放地址法（双重哈希）：ASL_成功 ≈ -ln(1-α)/α，ASL_失败 ≈ 1/(1-α)
- 实验验证：
  - 不同 α 值（0.1, 0.3, 0.5, 0.7, 0.9）下的实际 ASL
  - 与理论值对比
  - 绘制 α-ASL 曲线（Python matplotlib）

**验收标准**：
- 实际 ASL 与理论值误差 ≤ 10%
- α > 0.7 时性能明显退化（实验数据可验证）
- 提供完整的性能分析报告

### 3.4 哈希表扩容 / Rehash

**需求描述**：当装填因子超过阈值时自动扩容，保持 O(1) 均摊性能。

**实现要求**：

- 扩容触发条件：`α > load_factor_threshold`（默认 0.75）
- 扩容策略：
  - 容量翻倍（新容量 = 旧容量 × 2）
  - 新容量取最近的质数（减少冲突）
- Rehash 过程：
  - 分配新数组
  - 遍历旧表所有元素，重新计算哈希值插入新表
  - 释放旧数组
- 缩容（可选）：当 `α < 0.25` 时缩容为原来的一半
- 均摊分析：每次插入的均摊时间复杂度 O(1)

**验收标准**：
- 扩容后所有元素正确 rehash
- 扩容前后查找结果一致
- 均摊时间复杂度 O(1)
- 无内存泄漏
- 提供扩容过程的详细日志（旧容量 → 新容量、rehash 次数）

**技术参考**：
- [cppreference: std::unordered_map::rehash](https://en.cppreference.com/w/cpp/container/unordered_map/rehash) — C++ STL 的 rehash 接口
- [cppreference: std::unordered_map::max_load_factor](https://en.cppreference.com/w/cpp/container/unordered_map/max_load_factor) — C++ STL 的装填因子控制接口

### 3.5 LRU Cache 实现

**需求描述**：实现 Least Recently Used 缓存淘汰策略。

**实现要求**：

- 数据结构：哈希表 + 双向链表
  ```c
  typedef struct DListNode {
      int key;
      int value;
      struct DListNode *prev;
      struct DListNode *next;
  } DListNode;

  typedef struct {
      DListNode **hash_buckets;  // 哈希表：key → 链表节点
      DListNode *head;           // 双向链表头（最近使用）
      DListNode *tail;           // 双向链表尾（最久未使用）
      int size;
      int capacity;
  } LRUCache;
  ```
- 操作：
  - `get(key)`：O(1)，访问后移到链表头部
  - `put(key, value)`：O(1)，插入到链表头部，超出容量时淘汰尾部
  - `evict()`：淘汰最久未使用的元素
- 双向链表操作：
  - `remove(node)`：从链表中摘除
  - `addToHead(node)`：添加到链表头部
  - `moveToHead(node)`：移到链表头部（remove + addToHead）
  - `removeTail()`：移除链表尾部节点

**验收标准**：
- `get` 和 `put` 时间复杂度 O(1)
- 缓存满时正确淘汰 LRU 元素
- 访问已存在元素后其优先级更新
- 无内存泄漏
- 提供 LeetCode 146（LRU Cache）的解题验证

**技术参考**：
- LeetCode 146 是面试高频题，要求 O(1) 的 get/put
- C++ STL 可用 `std::unordered_map` + `std::list` 实现
- 参考 [cppreference: std::list](https://en.cppreference.com/w/cpp/container/list) — C++ 双向链表，`splice` 操作 O(1)

### 3.6 布隆过滤器（Bloom Filter）

**需求描述**：实现概率型数据结构，用于快速判断元素是否"可能存在"或"一定不存在"。

**实现要求**：

- 数据结构：
  ```c
  typedef struct {
      unsigned char *bits;   // 位数组
      int m;                 // 位数组长度
      int k;                 // 哈希函数个数
      HashFunc *hash_funcs;  // 哈希函数数组
  } BloomFilter;
  ```
- 操作：
  - `create(m, k)`：创建布隆过滤器
  - `add(key)`：添加元素（k 个哈希位置置 1）
  - `contains(key)`：查询元素（k 个哈希位置全为 1 则"可能存在"）
  - `destroy()`：销毁
- 参数优化：
  - 给定预期元素数 `n` 和误判率 `p`，计算最优 `m` 和 `k`
  - `m = -n * ln(p) / (ln2)²`
  - `k = (m/n) * ln2`
- 误判率分析：
  - 理论误判率：`p ≈ (1 - e^(-kn/m))^k`
  - 实验验证：插入 N 个元素后，测试 M 个不在集合中的元素的误判数

**验收标准**：
- 不存在假阴性（判断"不存在"则一定不存在）
- 假阳性率与理论值误差 ≤ 20%
- 参数优化公式正确
- 位数组使用位操作（`|`、`&`、`>>`），内存效率高
- 提供 LeetCode 面试题的解题验证

**技术参考**：
- [CLRS](https://mitpress.mit.edu/9780262046305/introduction-to-algorithms/) 习题 11.2-5
- 布隆过滤器原始论文：Bloom, "Space/Time Trade-offs in Hash Coding with Allowable Errors", 1970
- C++ `std::bitset` 可用于位数组实现：[cppreference: std::bitset](https://en.cppreference.com/w/cpp/utility/bitset)

### 3.7 一致性哈希（Consistent Hashing）

**需求描述**：实现分布式系统中的负载均衡算法。

**实现要求**：

- 基本一致性哈希：
  - 哈希环：将节点和键映射到 `[0, 2^32-1]` 的环形空间
  - 节点分配：键顺时针找到最近的节点
  - 数据结构：有序环（可用 C++ `std::map` 的 `lower_bound`）
- 虚拟节点：
  - 每个物理节点对应多个虚拟节点（如 150 个）
  - 解决数据倾斜问题
- 操作：
  - `addNode(node)`：添加节点，迁移受影响的数据
  - `removeNode(node)`：移除节点，迁移受影响的数据
  - `getNode(key)`：查找键所属节点
- 最小化数据迁移：添加/删除节点时仅影响相邻节点的数据

**验收标准**：
- 添加/删除节点时数据迁移量最小（仅影响 O(K/N) 的数据，K 为总键数，N 为节点数）
- 虚拟节点使数据分布均匀（标准差 ≤ 10%）
- 节点数变化时一致性保证（同一键在节点增减前后映射到相同或相邻节点）
- 提供 Python 模拟脚本验证数据分布均匀性

**技术参考**：
- Karger et al., "Consistent Hashing and Random Trees", STOC 1997
- [cppreference: std::map::lower_bound](https://en.cppreference.com/w/cpp/container/map/lower_bound) — 一致性哈希环的核心操作
- Amazon Dynamo、Apache Cassandra 均使用一致性哈希

## 4. 实现语言

### 4.1 C/C++ 核心实现

- **C 语言版本**：手动管理内存，手动实现链表/位数组，适合理解底层原理
- **C++ 版本**：使用 STL 容器（`std::vector`、`std::list`、`std::unordered_map`、`std::map`、`std::bitset`），代码更简洁安全
- **命名规范**：C 版本使用 `snake_case`，C++ 版本使用 `camelCase`
- **编译标准**：至少兼容 C++17，推荐 C++17

### 4.2 Python 辅助对比

- **正确性验证**：使用 Python `dict`/`set` 对比验证 C/C++ 实现的正确性

- **一致性哈希模拟**：使用 Python 模拟分布式场景

- Python 3.6 起 `dict` 使用紧凑字典（Compact Dict）实现，内存占用减少 20-25%
- Python 3.7 起保证 `dict` 插入顺序（作为语言特性保证，非实现细节）
- Python 3.12 起（PEP 709）字典推导式内联优化，性能提升约 2 倍
- Python 3.13 引入实验性自由线程模式（PEP 703，no-GIL），对 `dict` 并发访问有影响
- Python `dict` 使用开放地址法（二次探测的变体），装填因子阈值为 2/3
- 参考 [Python dict 源码](https://github.com/python/cpython/blob/main/Objects/dictobject.c)、[Python 3.12 What's New](https://docs.python.org/3/whatsnew/3.12.html)、[Python 3.13 What's New](https://docs.python.org/3/whatsnew/3.13.html)

**实现注意事项**：
- GNU libstdc++ 的 `std::unordered_map` 使用链地址法，每个桶为单链表
- LLVM libc++ 的 `std::unordered_map` 同样使用链地址法
- Boost.Unordered（Boost 1.80+）提供了基于开放地址法的 `boost::unordered_flat_map`，性能更优
- 参考 [cppreference: C++17](https://en.cppreference.com/w/cpp/17)、[cppreference: C++20](https://en.cppreference.com/w/cpp/20)

### 5.3 Python `dict` 实现演进详情

| Python 版本 | 变更 | 影响 |
|------------|------|------|
| 3.5 | `dict` 内部实现仍为稀疏数组 | 内存占用较大 |
| 3.6 | 紧凑字典（Compact Dict） | 内存减少 20-25%，插入顺序保留（CPython 实现细节） |
| 3.7 | 插入顺序保证 | 从实现细节升级为语言特性保证 |
| 3.9 | `dict` 合并运算符 `\|` / `\|=` | `d1 \| d2` 合并字典 |
| 3.12 | PEP 709 字典推导式内联 | 性能提升约 2 倍 |
| 3.13 | PEP 703 实验性自由线程 | no-GIL 模式下 `dict` 需要线程安全机制 |

**Python `dict` 冲突解决方式**：
- 使用开放地址法的变体（基于二次探测的扰动序列）
- 装填因子阈值为 2/3，超过时自动扩容
- 源码参考：[cpython/Objects/dictobject.c](https://github.com/python/cpython/blob/main/Objects/dictobject.c)

### 5.6 时效性注意事项

1. **C++ 标准演进**：
   - C++20 的 `contains()` 方法替代了 `find() != end()` 的惯用法，新代码应优先使用
   - C++17 的 `extract()`/`merge()` 允许节点级别的操作，避免不必要的拷贝
   - C++23 的 `std::flat_map`（基于有序数组）提供了另一种关联容器选择
   - 参考 [cppreference: C++23](https://en.cppreference.com/w/cpp/23)

2. **Python 版本演进**：
   - Python 3.12 的 PEP 709（字典推导式内联）改变了字典推导式的实现方式，性能提升但行为略有变化
   - Python 3.13 的 PEP 703（自由线程）引入 no-GIL 模式，`dict` 操作在多线程环境下需要额外注意
   - Python 3.10+ 的 `dict.union()` 和 `dict |` 运算符提供了新的合并方式
   - 参考 [Python 3.12 What's New](https://docs.python.org/3/whatsnew/3.12.html)、[Python 3.13 What's New](https://docs.python.org/3/whatsnew/3.13.html)

3. **布隆过滤器库**：
   - Python `pybloom-live` 已停止维护（最后更新 2017 年），不推荐使用
   - 推荐自行实现或使用 `bitarray` + 自定义哈希函数
   - C++ 可使用 `Boost.BloomFilter`（非官方）或自行实现

4. **Redis 渐进式 rehash**：
   - Redis 的哈希表扩容采用渐进式 rehash，将 rehash 分摊到后续操作中
   - 这是生产级优化，本项目基础实现可先使用一次性 rehash，后续作为进阶优化

## 7. 验收标准

### 7.1 代码质量验收

| 验收项 | 标准 |
|--------|------|
| 内存管理 | C 版本无内存泄漏（Valgrind/ASan 检测通过），C++ 版本使用 RAII |
| 命名规范 | C 版本 `snake_case`，C++ 版本 `camelCase`，同一文件内不混用 |
| 错误处理 | 检测空指针、满表、键不存在等异常情况 |
| 编译警告 | GCC `-Wall -Wextra -Wpedantic` 零警告 |
| 接口一致性 | 链地址法和开放地址法共享统一接口 |

### 7.2 功能验收

| 功能 | 验收标准 |
|------|---------|
| 链地址法 | 插入/查找/删除正确，无内存泄漏，ASL 与理论值误差 ≤ 10% |
| 线性探测 | 正确实现，出现一次聚集现象可统计验证 |
| 二次探测 | 正确实现，α < 0.5 时保证找到空槽 |
| 双重哈希 | 正确实现，冲突率低于线性/二次探测 |
| 扩容/rehash | 扩容后所有元素正确 rehash，均摊 O(1) |
| 全域哈希 | 冲突概率 ≤ 1/m，对抗性测试通过 |
| LRU Cache | get/put O(1)，LeetCode 146 通过 |
| 布隆过滤器 | 无假阴性，假阳性率与理论值误差 ≤ 20% |
| 一致性哈希 | 虚拟节点数据分布均匀（标准差 ≤ 10%），最小化数据迁移 |

### 7.3 性能验收

| 操作 | 链地址法 | 开放地址法 |
|------|---------|-----------|
| 查找（平均） | O(1 + α) | O(1 / (1-α)) |

| 删除（平均） | O(1 + α) | O(1 / (1-α)) |
| 扩容（均摊） | O(1) | O(1) |
