# 项目总览

> 时效性标注：本文档最后更新于 2026-05，技术栈版本信息基于 2025-05 检查结果。

## 项目简介

- **项目名称**：Data Structures and Algorithms（数据结构与算法）
- **目的**：系统学习数据结构与算法，提供完整的 C/C++ 实现
- **适用人群**：计算机专业学生、算法学习者、面试准备者
- **语言标准**：C17 / C++17 / Python 3.10+

### 目录设计意图

| 目录 | 设计意图 |
|------|---------|
| `src/` 按数据结构大类组织 | 遵循教科书的知识体系（线性表→栈→队列→树→图→哈希→堆），学习者可按认知梯度递进，每个大类对应一个独立 CMake 子模块 |
| `src/` 下每个子目录再按具体结构细分 | 如 `linear_list/` 下分 `sequential_list/`、`singly_linked_list/` 等，实现物理隔离，避免单目录文件膨胀，同时映射到 CMake 的 `add_subdirectory` 层级 |
| `advanced_algorithm/` 按算法范式组织 | 与数据结构模块按「结构」组织不同，高级算法按「范式」组织（DP、贪心、回溯、分治），因为同一范式下的不同问题共享相似的思维模式和解题框架 |
| `sorting/`、`searching/`、`string_algorithm/` 扁平组织 | 这些算法模块文件数量有限（4~8 个），且算法间无层级从属关系，扁平组织更直观 |
| `tests/` 独立于 `src/` | 测试代码与生产代码物理分离，测试通过 `#include` 或链接引用源码，避免测试逻辑混入发布构建 |
| `python/` 独立目录 | Python 脚本仅用于辅助验证（如排序对比可视化、DP 结果校验），不参与 CMake 构建，独立存放避免构建系统干扰 |

## 项目架构决策深度解析

### 1. C/C++ 分离原则

项目中 `.c` 文件和 `.cpp` 文件严格使用各自语言的标准库，绝不混用：

- **C 文件**（如 [circular_linked_list.c](../src/linear_list/circular_linked_list/circular_linked_list.c)）仅使用 `<stdio.h>`、`<stdlib.h>`、`<string.h>` 等 C 标准库头文件
- **C++ 文件**（如 [search_common.h](../src/searching/search_common.h)）使用 `<iostream>`、`<vector>`、`<algorithm>` 等 C++ 标准库头文件

**决策原因**：

1. **语言语义一致性**：C 和 C++ 是两种不同的语言。`<stdbool.h>` 在 C 中是必需的（C99 之前无原生布尔类型），但在 C++ 中 `bool` 是内建类型；`malloc/free` 与 `new/delete` 混用会导致未定义行为；C 的显式类型转换 `(Type*)` 与 C++ 的 `static_cast<Type*>` 语义不同。
2. **教学价值**：本项目面向学习者，C 文件展示底层内存管理（`malloc`/`free`、函数指针、手动泛型），C++ 文件展示现代抽象（RAII、模板、STL 容器），两种范式对照学习效果更佳。
3. **编译兼容性**：C 代码以 `-std=c17` 编译，C++ 代码以 `-std=c++17` 编译。混用头文件（如在 `.c` 文件中 `#include <iostream>`）会导致编译错误，因为 C 编译器无法解析 C++ 的名称修饰（name mangling）和类定义。
4. **分配原则**：基础数据结构（链表、栈、队列、图、哈希表、堆）使用 C 实现，强调手动内存管理和指针操作；高级数据结构（跳表、AVL 树、字典树）和算法（排序、搜索、字符串算法、DP）使用 C++ 实现，利用 STL 提升表达力。

### 2. CMake 多模块构建体系

项目采用分层 `add_subdirectory` 模式组织 CMake 构建，而非将所有目标定义在单个 `CMakeLists.txt` 中：

```
CMakeLists.txt                          # 根配置：设定 C/C++ 标准，add_subdirectory 引入各模块
├── src/linear_list/CMakeLists.txt      # 线性表模块：add_subdirectory 引入子模块
│   ├── sequential_list/CMakeLists.txt  #   顺序表：add_executable 定义可执行目标
│   ├── singly_linked_list/...
│   └── ...
├── src/stack/CMakeLists.txt
├── src/queue/CMakeLists.txt
├── tests/CMakeLists.txt                # 测试模块：定义 test_* 可执行目标
└── ...
```

**决策原因**：

1. **模块独立性**：每个子目录拥有独立的 `CMakeLists.txt`，新增或修改模块只需编辑该模块的构建文件，不会影响其他模块。例如在 `src/linear_list/skip_list/` 下新增测试文件，只需修改该目录的 `CMakeLists.txt`。
2. **构建粒度控制**：分层结构允许 CMake 仅重新配置变更的子目录，而非全量重新配置。对于包含 50+ 可执行目标的项目，这显著减少了增量构建时间。
3. **职责单一**：根 `CMakeLists.txt` 仅负责全局设置（语言标准、编译选项）和模块调度（`add_subdirectory`），不包含任何 `add_executable` 定义；叶子节点的 `CMakeLists.txt` 仅定义本模块的可执行目标，不关心全局配置。
4. **可扩展性**：新增数据结构模块只需创建目录、编写源码、添加 `CMakeLists.txt`，然后在父级 `CMakeLists.txt` 中添加一行 `add_subdirectory` 即可，无需修改根配置。

### 3. 头文件/源文件/入口文件三文件模式

C 模块普遍采用 `module.h` + `module.c` + `module_main.c` 三文件模式（如 [circular_linked_list.h](../src/linear_list/circular_linked_list/circular_linked_list.h) + [circular_linked_list.c](../src/linear_list/circular_linked_list/circular_linked_list.c) + [circular_linked_list_main.c](../src/linear_list/circular_linked_list/circular_linked_list_main.c)）；C++ 模块采用 `module.h` + `module.cpp` + `module_main.cpp` 模式（如 [skip_list.h](../src/linear_list/skip_list/skip_list.h) + [skip_list.cpp](../src/linear_list/skip_list/skip_list.cpp) + [skip_list_main.cpp](../src/linear_list/skip_list/skip_list_main.cpp)）。

**各文件职责**：

| 文件 | 职责 | 内容 |
|------|------|------|
| `module.h` | 接口声明 | 类型定义（`typedef`、`struct`、`class`）、函数原型、常量宏、头文件守卫 |
| `module.c` / `module.cpp` | 核心实现 | 包含自身头文件，实现所有声明的函数，不包含 `main` 函数 |
| `module_main.c` / `module_main.cpp` | 交互入口 | 包含模块头文件，定义 `main` 函数，提供菜单式交互界面 |

**决策原因**：

1. **关注点分离**：数据结构的实现逻辑（插入、删除、遍历）与用户交互逻辑（菜单、输入输出）属于不同抽象层次。分离后，实现文件可被测试文件和入口文件共同引用，避免重复编译。
2. **可测试性**：测试文件（如 [test_circular_queue.c](../tests/test_circular_queue.c)）通过 `#include` 引入 `.c` 源文件或链接 `.o` 目标文件来访问实现，而不需要 `main` 函数。如果实现和 `main` 在同一文件中，测试时会产生 `main` 冲突。
3. **CMake 集成**：三文件模式自然映射到 `add_executable` 命令。例如 `circular_linked_list` 目标定义为 `add_executable(circular_linked_list circular_linked_list_main.c circular_linked_list.c)`，入口文件和实现文件作为独立的编译单元参与构建。
4. **教学清晰度**：学习者可以先阅读 `.h` 了解「这个数据结构能做什么」，再阅读 `.c` 了解「它是怎么做的」，最后运行 `_main.c` 体验「它用起来是什么感觉」，形成从接口到实现到使用的认知闭环。

### 4. 自检测试框架设计思路

项目自建了轻量级测试框架 [self_test.h](../tests/self_test.h)，而非采用 Google Test、Catch2 等第三方框架。

**框架特点**：

- 纯宏实现，无运行时依赖，单头文件即可使用
- 提供 `ASSERT_EQ`、`ASSERT_TRUE`、`ASSERT_FALSE`、`ASSERT_NULL`、`ASSERT_NOT_NULL` 五个断言宏
- 提供 `TEST_BEGIN` / `TEST_END` 测试用例边界宏，自动统计通过/失败数
- 失败时输出文件名、行号、表达式和实际值，便于快速定位

**决策原因**：

1. **零依赖原则**：本项目是学习项目，引入 Google Test 需要配置 FetchContent 或 vcpkg，增加了构建复杂度。对于教学场景，测试框架本身不应成为学习门槛。
2. **C/C++ 双语兼容**：`self_test.h` 仅依赖 `<stdio.h>` 和 `<stdlib.h>`，C 和 C++ 测试文件均可直接使用。Google Test 仅支持 C++，无法用于 C 模块测试（如 [test_circular_queue.c](../tests/test_circular_queue.c)）。
3. **教学透明性**：测试框架的实现本身就是一个宏编程的教学案例。学习者可以打开 `self_test.h` 阅读全部实现，理解断言宏的 `do...while(0)` 惯用法、`#` 字符串化运算符、`__FILE__`/`__LINE__` 预定义宏等 C 语言核心机制。
4. **够用即可**：当前测试需求为功能正确性验证（断言值相等、指针非空），不涉及 mock、fixture、参数化测试等高级特性。自建框架的 7 个宏已覆盖全部需求。

### 5. CI/CD 三平台策略

项目使用 GitHub Actions 同时在 Ubuntu (GCC)、macOS (Clang)、Windows (MSVC) 三个平台上执行持续集成，配置文件为 [ci.yml](../.github/workflows/ci.yml)。

**决策原因**：

1. **编译器差异暴露**：三大编译器对标准的支持细节不同。GCC 对变长数组（VLA）的警告策略、Clang 对 `-Wextra` 的诊断覆盖范围、MSVC 对 `scanf` 的安全警告（`CRT_SECURE_NO_WARNINGS`）均有差异。三平台并行构建可以在 CI 阶段就发现这些兼容性问题。
2. **标准库实现差异**：GCC 使用 libstdc++，Clang 使用 libc++，MSVC 使用 MSVC STL。三者在 `std::vector` 的扩容策略、`std::sort` 的排序算法选择、异常处理行为等方面存在细微差别，跨平台验证确保代码不依赖特定实现的未定义行为。
3. **平台特定 API**：Windows 的路径分隔符、文件系统 API、终端编码（GBK vs UTF-8）与 Unix 系统不同。三平台测试确保交互式菜单（如 [circular_linked_list_main.c](../src/linear_list/circular_linked_list/circular_linked_list_main.c) 中的中文菜单）在各平台正常显示。
4. **学习者覆盖**：本项目的目标用户可能使用任意平台。三平台 CI 保证克隆仓库后 `cmake -B build && cmake --build build` 在任何平台上都能成功，降低上手门槛。

## 构建与运行

### 前置条件

| 工具 | 最低版本 | 安装方式 |
|------|---------|---------|
| CMake | 3.16+ | [官方下载](https://cmake.org/download/) 或包管理器安装 |
| GCC | 12+ | Linux: `sudo apt install gcc-12`；macOS: 通过 Homebrew 安装 |
| Clang | 15+ | macOS: Xcode Command Line Tools；Linux: `sudo apt install clang-15` |
| MSVC | 19.3+ | Windows: 安装 [Visual Studio 2022](https://visualstudio.microsoft.com/)（含 C++ 桌面开发工作负载） |

### CMake 构建

项目使用 CMake 的 out-of-source 构建模式（参见 [CMake 官方文档 - Running CMake](https://cmake.org/cmake/help/latest/manual/cmake.1.html#run-cmake-from-the-command-line)），构建产物生成在独立的 `build/` 目录中，不污染源码树。

```bash
# 第一步：配置（生成构建系统文件）
# CMake 会检测编译器、解析 CMakeLists.txt、生成 Makefile 或 .sln
cmake -B build

# 第二步：构建（编译所有目标）
cmake --build build

# 第三步（可选）：仅构建特定目标
cmake --build build --target circular_linked_list
cmake --build build --target searching
```

**配置选项说明**：

```bash
# 指定构建类型（Debug 保留调试信息，Release 启用优化）
cmake -B build -DCMAKE_BUILD_TYPE=Debug

# 指定编译器（覆盖自动检测）
cmake -B build -DCMAKE_C_COMPILER=gcc-12 -DCMAKE_CXX_COMPILER=g++-12

# Windows + MSVC 需先在 Developer Command Prompt 中运行
cmake -B build -G "Visual Studio 17 2022"
cmake --build build --config Release
```

> **参考**：CMake 构建系统详解参见 [CMake 官方文档 - Build Tool Mode](https://cmake.org/cmake/help/latest/manual/cmake.1.html#build-tool-mode) 和 [CMake Tutorial](https://cmake.org/cmake/help/latest/guide/tutorial/index.html)。

### 编译单个文件

无需 CMake，直接使用编译器编译单个源文件：

```bash
# 编译 C 文件
gcc -std=c17 -Wall -Wextra -o output src/path/to/file.c

# 编译 C++ 文件
g++ -std=c++17 -Wall -Wextra -o output src/path/to/file.cpp

# 编译含头文件的三文件模块
gcc -std=c17 -Wall -Wextra -I src/path/to/module -o output src/path/to/module/module_main.c src/path/to/module/module.c
```

### 运行自检测试

```bash
# 方式一：通过 CMake 运行（推荐）
cmake --build build --target test_circular_queue
./build/test_circular_queue

cmake --build build --target test_sorting
./build/test_sorting

# 方式二：手动编译运行
gcc tests/test_circular_queue.c src/queue/circular_queue/circular_queue.c -I tests -o test_cq
./test_cq

g++ tests/test_sorting.cpp src/sorting/sequential_list_sorting.cpp -I tests -o test_sort
./test_sort
```

## 测试体系

项目采用轻量级自检测试框架，所有测试宏定义在 [self_test.h](../tests/self_test.h) 中。

### 测试宏一览

| 宏 | 功能 | 实现原理 |
|----|------|---------|
| `ASSERT_EQ(a, b)` | 断言两值相等 | 比较 `(a) == (b)`，失败时输出实际值与期望值 |
| `ASSERT_TRUE(expr)` | 断言表达式为真 | 评估 `expr`，失败时输出表达式文本 |
| `ASSERT_FALSE(expr)` | 断言表达式为假 | 评估 `!(expr)`，失败时输出表达式文本 |
| `ASSERT_NULL(ptr)` | 断言指针为 NULL | 比较 `(ptr) == NULL` |
| `ASSERT_NOT_NULL(ptr)` | 断言指针非 NULL | 比较 `(ptr) != NULL` |
| `TEST_BEGIN` | 测试用例开始标记 | 重置通过/失败/总数计数器 |
| `TEST_END` | 测试用例结束标记 | 输出统计结果，返回失败数（0 表示全部通过） |

### 测试文件

| 测试文件 | 测试目标 | 语言 | 被测源文件 |
|---------|---------|------|-----------|
| [test_circular_queue.c](../tests/test_circular_queue.c) | 循环队列 | C | [circular_queue.c](../src/queue/circular_queue/circular_queue.c) |
| [test_hash_table.c](../tests/test_hash_table.c) | 哈希表 | C | [hash_table_chaining.c](../src/hash_table/hash_table_chaining.c)、[hash_table_open_addressing.c](../src/hash_table/hash_table_open_addressing.c) |
| [test_sorting.cpp](../tests/test_sorting.cpp) | 排序算法 | C++ | [sequential_list_sorting.cpp](../src/sorting/sequential_list_sorting.cpp) |
| [test_searching.cpp](../tests/test_searching.cpp) | 搜索算法 | C++ | [binary_search.cpp](../src/searching/binary_search.cpp) 等 |

## 持续集成

项目使用 GitHub Actions 实现三平台持续集成，配置文件：[ci.yml](../.github/workflows/ci.yml)

| 平台 | 编译器 | 运行环境 | 构建步骤 |
|------|--------|---------|---------|
| Ubuntu | GCC 12 | `ubuntu-latest` | apt 安装 GCC → cmake 配置 → cmake 构建 → 运行测试 |
| macOS | Clang (Xcode) | `macos-latest` | xcode-select → brew 安装 CMake → cmake 配置 → cmake 构建 → 运行测试 |
| Windows | MSVC 19 | `windows-latest` | msvc-dev-cmd → choco 安装 CMake → cmake 配置 → cmake 构建 → 运行测试 |

三个 Job 并行执行，任一平台构建失败即标记 CI 不通过。
