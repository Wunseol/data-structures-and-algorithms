# 09 字符串算法模块需求文档

## 1. 模块概述

字符串算法是数据结构与算法中极为重要的领域，广泛应用于文本编辑器、搜索引擎、生物信息学（DNA 序列比对）、信息安全（入侵检测）、数据压缩等场景。与数值型数据不同，字符串具有独特的结构特征（有序字符序列、可变长度、模式匹配需求），因此需要专门的算法来处理。

本模块为**全新模块**，项目中当前无任何字符串算法实现。本模块将从基础的字符串操作出发，逐步覆盖字符串匹配算法、Trie 树、后缀数组、最长公共子序列、最长回文子串、字符串哈希和正则表达式基础等核心内容。

**模块目标：**
- 掌握字符串匹配算法从暴力到 KMP 的演进过程
- 理解并实现 Trie 树（与树模块交叉引用）
- 了解后缀数组/后缀树的基本概念
- 掌握最长公共子序列算法（与动态规划模块交叉引用）
- 掌握最长回文子串算法（Manacher 算法）
- 理解字符串哈希（Rolling Hash）的原理与应用
- 了解正则表达式的基本原理
- 熟悉 C/C++ 和 Python 中的字符串处理工具

**实现语言：** C/C++（核心算法实现），Python（字符串处理与验证，Python 拥有优秀的字符串支持）

## 3. 新增需求

### 3.1 字符串匹配算法

#### 3.1.1 暴力匹配（Brute Force / Naive Match）

- **原理：** 逐个位置尝试匹配模式串，失配时模式串右移一位重新开始
- **时间复杂度：** 最好 O(n+m)，最坏 O(n×m)（n 为文本长度，m 为模式长度）
- **空间复杂度：** O(1)
- **实现要求：**
  - 基本暴力匹配实现
  - 统计比较次数
  - 构造最坏情况输入（如文本 "AAAA...A"，模式 "AAAAB"）并分析
- **学习重点：** 理解暴力匹配的效率瓶颈——回溯导致重复比较

#### 3.1.2 KMP 算法（Knuth-Morris-Pratt）

- **原理：** 利用已匹配信息，通过 next 数组（前缀函数/部分匹配表）避免回溯
- **时间复杂度：** O(n+m)
- **空间复杂度：** O(m)（next 数组）
- **实现要求：**
  - next 数组的计算（两种版本：原始 next 和优化版 nextval）
  - KMP 匹配过程
  - 统计比较次数，与暴力匹配对比
  - 详细注释 next 数组的构造过程
- **学习重点：**
  - 理解"前缀"和"后缀"的概念
  - 理解 next 数组的含义：`next[j]` 表示 `P[0..j-1]` 的最长相等前后缀长度
  - 理解 nextval 的优化：当 `P[j] == P[next[j]]` 时进一步跳过
  - 能手动模拟 KMP 的执行过程

**next 数组计算示例：**

```
模式串 P:    a b a b a a a b a
下标 j:      0 1 2 3 4 5 6 7 8
next[j]:    -1 0 0 1 2 3 1 1 2
nextval[j]: -1 0-1 0-1 3 1 0 2
```

#### 3.1.3 Rabin-Karp 算法

- **原理：** 使用哈希函数对模式串和文本子串进行哈希比较，哈希值相同时再逐字符比较
- **时间复杂度：** 平均 O(n+m)，最坏 O(n×m)（哈希冲突多时）
- **空间复杂度：** O(1)
- **实现要求：**
  - 基本数值哈希版本
  - Rolling Hash 优化（O(1) 时间计算滑动窗口哈希）
  - 处理哈希冲突的策略
  - 多模式匹配扩展（选做）
- **学习重点：**
  - 理解 Rolling Hash 的核心思想：利用前一个窗口的哈希值 O(1) 计算下一个窗口
  - 理解模运算在哈希中的作用
  - 理解选择合适质数作为模数的重要性

#### 3.1.4 Boyer-Moore 算法

- **原理：** 从模式串末尾开始比较，利用"坏字符规则"和"好后缀规则"进行大步跳跃
- **时间复杂度：** 最好 O(n/m)（每次跳跃整个模式长度），最坏 O(n×m）
- **空间复杂度：** O(m + |Σ|)（Σ 为字符集大小）
- **实现要求：**
  - 坏字符规则（Bad Character Rule）的实现
  - 好后缀规则（Good Suffix Rule）的实现（选做，实现较复杂）
  - 仅使用坏字符规则的简化版本
  - 统计比较次数，与其他算法对比
- **学习重点：**
  - 理解"从右向左比较"的逆向思维
  - 理解坏字符规则如何决定跳跃距离
  - 理解 Boyer-Moore 在实际应用中（如 grep）为何通常最快
  - 注意：C++17 的 `std::boyer_moore_searcher` 和 `std::boyer_moore_horspool_searcher` 基于 BM 和 BMH 算法

### 3.2 Trie 树（字典树/前缀树）

- **原理：** 利用字符串的公共前缀构建多叉树，实现高效的字符串插入和查找
- **时间复杂度：** 插入 O(m)，查找 O(m)（m 为字符串长度）
- **空间复杂度：** O(N×M)（N 为字符串数量，M 为平均长度）
- **实现要求：**
  - 基本 Trie 树的插入、查找、删除
  - 前缀搜索（查找所有以给定前缀开头的字符串）
  - 统计以给定前缀开头的字符串数量
  - 使用数组实现和哈希表实现两种版本
- **学习重点：**
  - 理解 Trie 树利用公共前缀节省空间的思想
  - 理解 Trie 树与 BST 在字符串查找上的效率差异
  - 交叉引用：Trie 树属于树结构，与树模块（`04-tree.md`）建立交叉引用
- **应用场景：** 自动补全、拼写检查、IP 路由表、词频统计

### 3.3 后缀数组与后缀树基础

#### 3.3.1 后缀数组（Suffix Array）

- **原理：** 将字符串的所有后缀按字典序排序，排序后的起始位置数组即为后缀数组
- **构建时间：** O(nlogn)（基于倍增法）或 O(n)（SA-IS 算法）
- **实现要求：**
  - 理解后缀数组的概念和构建过程
  - 实现基于排序的朴素构建方法 O(n²logn)
  - 实现倍增法构建 O(nlog²n) 或 O(nlogn)
  - 实现 LCP 数组（最长公共前缀数组）的计算
  - 不要求实现 SA-IS 线性算法（过于复杂）
- **学习重点：**
  - 理解后缀数组与后缀树的关系
  - 理解 LCP 数组在解决字符串问题中的作用
  - 应用：最长重复子串、不同子串计数

#### 3.3.2 后缀树（Suffix Tree）

- **原理：** 压缩后的 Trie 树，将所有后缀存储在树中
- **实现要求：**
  - **不要求实现**，仅要求理解概念和基本结构
  - 理解后缀树与后缀数组的关系
  - 理解 Ukkonen 算法的基本思想（O(n) 在线构建）
  - 了解后缀自动机（Suffix Automaton）的存在
- **学习重点：** 后缀树是字符串算法中最复杂的数据结构之一，理解其概念即可

### 3.4 最长公共子序列（LCS）

- **原理：** 动态规划，`dp[i][j]` 表示 `A[0..i-1]` 和 `B[0..j-1]` 的最长公共子序列长度
- **时间复杂度：** O(n×m)
- **空间复杂度：** O(n×m)，可优化至 O(min(n,m))
- **实现要求：**
  - LCS 长度计算
  - LCS 具体序列的回溯输出
  - 空间优化版本
  - 统计比较次数
- **学习重点：**
  - 理解子序列与子串的区别（子序列不要求连续，子串要求连续）
  - 理解 DP 状态转移方程：
    - `dp[i][j] = dp[i-1][j-1] + 1`（当 `A[i-1] == B[j-1]`）
    - `dp[i][j] = max(dp[i-1][j], dp[i][j-1])`（当 `A[i-1] != B[j-1]`）
  - 交叉引用：LCS 属于动态规划，与高级算法模块（`10-advanced-algorithms.md`）建立交叉引用
- **应用场景：** diff 工具、DNA 序列比对、版本控制

### 3.5 最长回文子串

#### 3.5.1 暴力法

- **时间复杂度：** O(n³)
- **实现要求：** 作为基准对比

#### 3.5.2 中心扩展法

- **时间复杂度：** O(n²)
- **空间复杂度：** O(1)
- **实现要求：**
  - 奇数长度和偶数长度回文分别处理
  - 统计比较次数

#### 3.5.3 Manacher 算法

- **原理：** 利用已计算的回文信息，避免重复计算，实现线性时间
- **时间复杂度：** O(n)
- **空间复杂度：** O(n)
- **实现要求：**
  - 字符串预处理（插入分隔符统一奇偶长度）
  - Manacher 核心算法实现
  - 详细注释算法的每一步
- **学习重点：**
  - 理解"利用对称性避免重复计算"的核心思想
  - 理解 P 数组（回文半径数组）的含义
  - 理解当前最右回文边界 R 和中心 C 的维护
  - 能手动模拟 Manacher 算法的执行过程

### 3.6 字符串哈希（Rolling Hash）

- **原理：** 将字符串映射为数值，利用多项式哈希实现 O(1) 的子串哈希计算
- **哈希函数：** `H(s) = (s[0]×b^(n-1) + s[1]×b^(n-2) + ... + s[n-1]) mod M`
- **子串哈希：** `H(s[l..r]) = (H(s[0..r]) - H(s[0..l-1])×b^(r-l+1)) mod M`
- **实现要求：**
  - 多项式滚动哈希实现
  - 子串哈希 O(1) 查询
  - 双哈希（使用两个不同的质数）降低冲突概率
  - 应用：快速判断两个子串是否相等、最长重复子串
- **学习重点：**
  - 理解多项式哈希的数学原理
  - 理解模运算的性质
  - 理解双哈希降低冲突的原理
  - 理解字符串哈希与 Rabin-Karp 的关系
- **应用场景：** 快速字符串比较、去重、Rabin-Karp 匹配

### 3.7 正则表达式基础

- **实现要求：** 不要求实现正则表达式引擎，但需理解基本原理
- **学习内容：**
  - 正则表达式的基本语法：字符类、量词、锚点、分组
  - 正则表达式的匹配原理：NFA（非确定性有限自动机）和 DFA（确定性有限自动机）
  - 回溯匹配与 NFA 的关系
  - 贪婪匹配 vs 非贪婪匹配
  - Python `re` 模块的使用
  - C++ `<regex>` 库的使用（注意性能问题）
- **学习重点：**
  - 理解正则表达式与有限自动机的等价性
  - 理解回溯爆炸问题（ReDoS）及其防范
  - 实际编程中使用正则表达式的最佳实践

## 4. 实现语言

### 4.1 C/C++ 实现策略

C/C++ 用于核心算法的实现，重点在于理解底层机制和手动内存管理。

**C 语言字符串处理：**

| 函数 | 头文件 | 说明 | 安全替代 |
|------|--------|------|---------|
| `strlen` | `<string.h>` | 计算字符串长度 | `strnlen_s`（C11 Annex K） |
| `strcmp` | `<string.h>` | 字符串比较 | `strcmp_s`（C11 Annex K） |
| `strncmp` | `<string.h>` | 限定长度比较 | `strncmp_s`（C11 Annex K） |
| `strcpy` | `<string.h>` | 字符串复制 | `strcpy_s`（C11 Annex K） |
| `strncpy` | `<string.h>` | 限定长度复制 | `strncpy_s`（C11 Annex K） |
| `strcat` | `<string.h>` | 字符串连接 | `strcat_s`（C11 Annex K） |
| `strstr` | `<string.h>` | 子串查找 | 无直接安全替代 |
| `strchr` | `<string.h>` | 字符查找 | 无直接安全替代 |

> **时效性说明：**
> - C11 标准引入了 Annex K "Bounds-checking interfaces"，提供了 `strcpy_s`、`strncpy_s`、`strcat_s`、`strncat_s`、`strcmp_s`、`strncmp_s` 等安全函数
> - **重要：** Annex K 是可选（optional）的，并非所有编译器都实现。GCC 和 Clang 默认不实现 Annex K，MSVC 实现了部分
> - C23 标准草案中 Annex K 仍为可选，但推荐实现
> - 在教学代码中，建议至少了解这些安全函数的存在，实际项目中优先使用安全版本

**C++ 字符串处理：**

| 组件 | 头文件 | 说明 | 标准版本 |
|------|--------|------|---------|
| `std::string` | `<string>` | 动态字符串，自动内存管理 | C++98 |
| `std::string_view` | `<string_view>` | 非拥有字符串视图，避免不必要的拷贝 | **C++17** |
| `std::getline` | `<string>` | 读取整行 | C++98 |
| `std::stoi` / `std::stol` / `std::stod` | `<string>` | 字符串转数值 | C++11 |
| `std::to_string` | `<string>` | 数值转字符串 | C++11 |
| `std::regex` | `<regex>` | 正则表达式 | C++11 |

> **时效性说明：**
> - C++17 引入的 `std::string_view` 是重要的性能优化工具，它是一个轻量级的非拥有字符串引用，避免了 `std::string` 构造和拷贝的开销
> - C++20 引入了 `std::format`（基于 `<format>` 头文件），替代 `sprintf` 和 `std::stringstream` 进行格式化输出
> - C++20 为 `std::string` 添加了 `starts_with()` 和 `ends_with()` 方法
> - C++20 引入了 `std::u8string`（UTF-8 字符串），`char8_t` 类型
> - C++23 为 `std::string` 添加了 `contains()` 方法，为 `std::string_view` 添加了 `contains()` 方法
> - **注意：** C++ 的 `std::regex` 实现性能较差（特别是 GCC 的实现），在性能敏感场景建议使用第三方库如 [re2](https://github.com/google/re2) 或 [PCRE2](https://github.com/PCRE2Project/pcre2)

### 4.2 Python 实现策略

## 5. 学习路径

### 第一阶段：字符串基础与暴力匹配（1 周）

```
C/C++ 字符串操作复习 → 暴力匹配 → 性能分析
```

**学习重点：**
- 复习 C 语言 `string.h` 函数和 C++ `std::string` 的使用
- 理解字符串的存储方式（C 风格字符串 vs `std::string` vs `std::string_view`）
- 实现暴力匹配，构造最坏情况输入
- 练习：使用 `strstr` / `std::string::find` 与自实现暴力匹配对比

### 第二阶段：KMP 与字符串匹配进阶（2-3 周）

```
KMP 算法 → Rabin-Karp → Boyer-Moore → 匹配算法对比
```

**学习重点：**
- KMP：深入理解 next 数组的构造，手动模拟匹配过程
- Rabin-Karp：理解 Rolling Hash 原理，理解哈希冲突处理
- Boyer-Moore：理解坏字符规则，理解从右向左比较的逆向思维
- 对比：三种算法的适用场景（KMP 适合单模式精确匹配，Rabin-Karp 适合多模式/哈希应用，BM 适合实际文本搜索）
- 练习：LeetCode 28（strStr）、LeetCode 214（最短回文）

### 第三阶段：字符串数据结构（2 周）

```
Trie 树 → 后缀数组 → 后缀树概念
```

**学习重点：**
- Trie 树：实现插入、查找、前缀搜索，理解公共前缀优化
- 后缀数组：理解构建过程和 LCP 数组，应用：最长重复子串
- 后缀树：理解概念，不要求实现
- 练习：LeetCode 208（实现 Trie）、LeetCode 212（单词搜索 II）

### 第四阶段：字符串 DP 与高级算法（2 周）

```
LCS → 最长回文子串 → Manacher → 字符串哈希
```

**学习重点：**
- LCS：理解 DP 状态转移，掌握空间优化技巧
- 最长回文子串：从暴力到中心扩展到 Manacher 的优化过程

## 7. 验收标准

### 7.1 字符串匹配算法验收

| 验收项 | 标准 | 验证方法 |
|--------|------|---------|
| 暴力匹配 | 对随机文本和模式串，结果与 `strstr` 一致 | 对比测试 |
| KMP | 对随机文本和模式串，结果与暴力匹配一致；比较次数少于暴力匹配 | 对比测试 + 统计验证 |
| KMP next 数组 | 手动计算的 next 数组与程序输出一致 | 特定模式串测试 |
| KMP nextval 优化 | nextval 数组正确，且匹配时比较次数不多于使用 next 的版本 | 对比测试 |
| Rabin-Karp | 结果与暴力匹配一致 | 对比测试 |
| Boyer-Moore | 结果与暴力匹配一致；在典型文本上比较次数少于 KMP | 对比测试 + 统计验证 |
