## 2 语言标准

### 2.1 C17（ISO/IEC 9899:2018）

#### 概述

C17 是 C 语言的当前国际标准，正式编号为 ISO/IEC 9899:2018，于 2018 年 6 月发布。C17 是对 C11 标准的缺陷修复版本（Defect Report），不引入新的语言特性，仅修正了 C11 中发现的约 50 余项技术勘误和歧义。因此 C17 也常被称为"C18"（按发布年份）或"C11+DR"。

C17 相较于 C11 的关键修正包括：

- 修正了 `_Atomic` 限定符在某些上下文中的未定义行为
- 明确了 `alignas` 与 `_Alignas` 的语义一致性
- 修复了边界检查函数（Annex K）的规范歧义
- 移除了 C11 中标记为过时的 `gets` 函数（`gets` 在 C11 中已被标记为过时，C17 正式从标准中删除）

C17 中未引入新特性这一事实，使其成为目前最稳定的 C 语言标准版本，非常适合教学项目使用——既不会因新特性带来额外认知负担，又能确保代码在所有主流编译器上获得一致的行为。

#### 官方文档

- [ISO/IEC 9899:2018 标准购买页面](https://www.iso.org/standard/74528.html)
- [cppreference C 语言参考](https://en.cppreference.com/w/c)
- [WG14 N2176（C17 最终草案）](http://www.open-std.org/jtc1/sc22/wg14/www/abq/c17_updated_proposed_fdis.pdf)

#### 本项目使用方式

在根 [CMakeLists.txt](file:///d:/Dev/DevWorkSpace/VS%20Code/data-structures-and-algorithms/CMakeLists.txt) 中通过以下配置指定 C17 标准：

```cmake
set(CMAKE_C_STANDARD 17)
set(CMAKE_C_STANDARD_REQUIRED ON)
```

- `CMAKE_C_STANDARD 17`：告知 CMake 编译 C 源文件时应使用 C17 标准。CMake 会自动为不同编译器生成对应的标准选项（如 GCC/Clang 的 `-std=c17`，MSVC 的 `/std:c17`）。
- `CMAKE_C_STANDARD_REQUIRED ON`：将标准要求设为强制。若编译器不支持 C17，CMake 配置阶段将直接报错而非静默降级。这避免了因编译器回退到旧标准而导致的隐蔽兼容性问题。

项目中 C 语言实现涵盖的模块包括：线性表（顺序表、单链表、双链表、循环链表）、栈与队列（链栈、循环队列、链队列、优先队列）、图（邻接矩阵、邻接表、并查集）、哈希表（链地址法、开放寻址法）、堆（最大堆、最小堆）等。这些模块以 `.c` / `.h` 文件组织，均遵循 C17 标准。

| 编译器 | C17 最低支持版本 | 备注 |
|--------|------------------|------|
| GCC | 8.0 | GCC 8 起完整支持 C17（`-std=c17`），GCC 5-7 通过 `-std=c1x` 或 `-std=gnu1x` 部分支持 |
| Clang | 6.0 | Clang 6 起支持 `-std=c17`，更早版本通过 `-std=c1x` 部分支持 |
| MSVC | 19.27（VS 2019 16.7） | MSVC 对 C 标准的支持历来滞后，`/std:c17` 从 VS 2019 16.7 起可用 |

#### 变更风险标注

⚠️ **低风险** — C17 作为 ISO 发布的正式标准，其语义不会变更。但需关注以下趋势：

- **C23（ISO/IEC 9899:2024）** 已于 2024 年发布，引入了若干重要特性：`nullptr` 常量、`bool`/`true`/`false` 成为关键字、`constexpr` 支持、改进的属性语法（`[[deprecated]]` 等）、位精确整数类型（`_BitInt`）以及 `#embed` 预处理指令等。C23 是下一个重大标准版本。
- GCC 14+、Clang 18+ 已提供 C23 实验性支持（`-std=c23`），MSVC 尚未宣布 C23 支持计划。
- 本项目暂不计划升级至 C23，因为 C23 的编译器支持尚不成熟，且教学项目更侧重稳定性。

### 2.2 C++17（ISO/IEC 14882:2020）

#### 概述

C++17 是 C++ 语言的当前国际标准，正式编号为 ISO/IEC 14882:2020（注意：标准编号中的年份是发布年份 2020，但技术内容通常被称为"C++17"，因其最终草案完成于 2017 年）。C++17 是继 C++14 之后的一次重大更新，引入了大量实用特性。

C++17 的核心新特性包括：

- **结构化绑定（Structured Bindings）**：`auto [key, value] = *it;`，简化对 `pair`、`tuple`、结构体的解构
- **`std::optional<T>`**：表示"可能没有值"的语义，替代裸指针或哨兵值
- **`std::variant<T...>`**：类型安全的联合体，替代传统 `union`
- **`std::any`**：可持有任意类型值的类型安全容器
- **`std::string_view`**：非拥有式字符串引用，避免不必要的字符串拷贝
- **`if constexpr`**：编译期条件编译，替代 SFINAE 实现编译期分支
- **折叠表达式（Fold Expressions）**：简化变参模板的递归展开
- **`[[nodiscard]]`、`[[maybe_unused]]`、`[[fallthrough]]`** 属性：增强编译器对代码意图的理解
- **内联变量（Inline Variables）**：头文件中定义常量不再需要 ODR 问题规避
- **`std::filesystem`**：文件系统操作标准化
- **并行 STL 算法**：`std::execution::par` 策略支持算法并行化
- **类模板参数推导（CTAD）**：`std::pair(1, 2.0)` 无需显式指定模板参数

C++17 未纳入但曾广泛讨论的特性包括：Concepts（推迟至 C++20）、Modules（推迟至 C++20）、Ranges（推迟至 C++20）、协程（推迟至 C++20）。

#### 官方文档

- [ISO/IEC 14882:2020 标准购买页面](https://www.iso.org/standard/79358.html)
- [cppreference C++ 语言参考](https://en.cppreference.com/w/cpp)
- [C++17 特性概览（cppreference）](https://en.cppreference.com/w/cpp/17)

#### 本项目使用方式

在根 [CMakeLists.txt](file:///d:/Dev/DevWorkSpace/VS%20Code/data-structures-and-algorithms/CMakeLists.txt) 中通过以下配置指定 C++17 标准：

```cmake

set(CMAKE_CXX_STANDARD_REQUIRED ON)
```

- `CMAKE_CXX_STANDARD 17`：告知 CMake 编译 C++ 源文件时应使用 C++17 标准。CMake 会为不同编译器生成对应选项（如 GCC/Clang 的 `-std=c++17`，MSVC 的 `/std:c++17`）。
- `CMAKE_CXX_STANDARD_REQUIRED ON`：强制要求 C++17 支持，编译器不支持时直接报错。

项目中 C++ 实现涵盖的模块包括：高级算法（回溯、分治、动态规划、贪心）、搜索算法（二分搜索、插值搜索、斐波那契搜索）、排序算法（桶排序、计数排序、基数排序）、字符串算法（KMP、Boyer-Moore、Rabin-Karp、Manacher、LCS、滚动哈希）、树结构（AVL 树、Huffman 树、Trie）、位运算、数学算法、双端队列、跳表等。这些模块以 `.cpp` / `.h` 文件组织，均遵循 C++17 标准。

选择 C++17 而非 C++20 的原因：

1. **编译器支持成熟度**：C++17 在 GCC 12+、Clang 15+、MSVC 19.3+ 上均有完整且经过充分验证的支持；C++20 的部分特性（如 Modules、Concepts）在不同编译器间仍存在实现差异。
2. **跨平台一致性**：本项目需要在 GCC、Clang、MSVC 三种编译器上构建，C++17 能保证三端行为一致。
3. **教学适用性**：C++17 提供了 `std::optional`、`std::string_view`、结构化绑定等现代特性，足以编写简洁优雅的代码，同时避免 C++20 Concepts/Ranges 等高级特性对初学者造成的认知负担。

#### 版本兼容性说明

| 编译器 | C++17 最低支持版本 | 备注 |
|--------|---------------------|------|
| GCC | 7.1 | GCC 7 起基本支持 C++17，GCC 8 完整支持（含 `std::filesystem`） |
| Clang | 5.0 | Clang 5 起基本支持，Clang 7 完整支持（含并行算法需配合 Intel TBB） |
| MSVC | 19.1（VS 2017 15.3） | MSVC 19.1 起支持 `/std:c++17`，19.14 起完整支持 `std::filesystem` |

本项目 CI 要求 GCC 12+、Clang 15+（macOS Xcode）、MSVC 19.3+，均远超 C++17 最低要求。

#### 变更风险标注

⚠️ **低风险** — C++17 作为 ISO 正式标准，语义不会变更。但需关注：

- **C++20（ISO/IEC 14882:2020）** 已发布，引入 Concepts、Ranges、Modules、Coroutines、`std::format` 等重大特性。C++20 是 C++ 语言演进的重要里程碑。
- **C++23（ISO/IEC 14882:2024）** 已发布，引入 `std::print`、`std::expected`、`std::flat_map`/`std::flat_set`、显式 `this` 参数（推导 this）等特性。
- GCC 12+ 已提供较好的 C++20 支持，GCC 14+ 支持 C++23 部分特性。MSVC 从 VS 2019 16.10 起支持 C++20 大部分特性。
- 本项目暂不升级至 C++20/23，但未来可根据需要逐步引入，如 Concepts 可改善模板错误信息可读性。

## 3 构建系统

### 3.1 CMake 3.16+

#### 概述

CMake 是一个开源的跨平台构建系统生成器，由 Kitware 公司开发维护。CMake 本身不直接编译代码，而是根据 `CMakeLists.txt` 配置文件生成特定构建工具的原生构建文件（如 Unix Makefile、Ninja 构建文件、Visual Studio 解决方案、Xcode 项目等），再由底层构建工具执行实际编译。

CMake 3.16 发布于 2019 年 11 月，选择此版本作为最低要求的原因：

1. **预编译头文件支持（`target_precompile_headers`）**：CMake 3.16 引入了预编译头文件的原生支持命令，可显著加速 C++ 项目的编译速度。
2. **广泛可用性**：CMake 3.16 已被 Ubuntu 20.04 LTS、Debian Bullseye 等长期支持发行版的官方仓库收录，用户无需手动安装即可获取。
3. **稳定性**：3.16 是一个成熟的 LTS 级别版本，所有主流特性均已稳定。

#### 官方文档

- [CMake 官方文档](https://cmake.org/documentation/)
- [CMake 3.16 Release Notes](https://cmake.org/cmake/help/latest/release/3.16.html)
- [CMake Tutorial](https://cmake.org/cmake/help/latest/guide/tutorial/index.html)

#### 本项目使用方式

根 [CMakeLists.txt](file:///d:/Dev/DevWorkSpace/VS%20Code/data-structures-and-algorithms/CMakeLists.txt) 的完整配置解析：

```cmake
cmake_minimum_required(VERSION 3.16)
```

`cmake_minimum_required` 指定项目所需的最低 CMake 版本。此命令必须位于 `CMakeLists.txt` 的第一行（注释除外）。设置 `VERSION 3.16` 后：
- 若用户安装的 CMake 版本低于 3.16，CMake 将在配置阶段立即报错并终止。
- CMake 会将策略（Policy）兼容性设置为 3.16 对应的行为，确保后续版本中行为变更的策略不会影响已有项目。

```cmake
project(data_structures_and_algorithms LANGUAGES C CXX)
```

`project` 命令定义项目名称和支持的编程语言：
- `data_structures_and_algorithms`：项目名称，此名称将影响默认的输出目标名称和变量 `PROJECT_NAME`。
- `LANGUAGES C CXX`：声明项目使用 C 和 C++ 两种语言。CMake 将据此检测系统中可用的 C 和 C++ 编译器。若省略 `LANGUAGES`，默认为 `C CXX`，但显式声明更清晰。若某种语言的编译器未找到，配置将失败。

```cmake
set(CMAKE_C_STANDARD 17)
set(CMAKE_C_STANDARD_REQUIRED ON)
set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
```

这四行设置语言标准（详见第 2 节）。`CMAKE_C_STANDARD` 和 `CMAKE_CXX_STANDARD` 是 CMake 的内置变量，影响所有后续目标的编译标准选项。设置为全局变量而非逐目标设置（`target_compile_features`）的方式更简洁，适合本项目所有模块使用相同标准的场景。

```cmake
add_compile_options(-Wall -Wextra)
```

`add_compile_options` 向所有目标添加编译器选项：
- `-Wall`：启用编译器的大部分警告。名称中的"all"具有误导性，实际上 `-Wall` 并未启用所有警告，而是一组经过筛选的、误报率较低的警告集合。
- `-Wextra`：启用 `-Wall` 未涵盖的额外警告，如未使用的函数参数、比较有符号与无符号整数等。

选择 `-Wall -Wextra` 而非更严格的 `-Wpedantic` 或 `-Werror` 的原因：教学项目需要容许一定程度的代码探索，`-Werror`（将警告视为错误）可能阻碍初学者的实验性编码。`-Wall -Wextra` 在提供有用警告与保持编译通过之间取得了良好平衡。

> **注意**：`-Wall -Wextra` 是 GCC/Clang 特有的选项。MSVC 使用不同的警告选项体系（如 `/W4`），但 CMake 会在 MSVC 下自动忽略不识别的选项，不会导致构建失败。若需跨平台精确控制警告级别，应使用条件判断分别设置。

```cmake
add_subdirectory(src/linear_list)
add_subdirectory(src/stack)
add_subdirectory(src/queue)
add_subdirectory(src/tree)
add_subdirectory(src/graph)
add_subdirectory(src/hash_table)
add_subdirectory(src/heap)
add_subdirectory(src/sorting)
add_subdirectory(src/searching)
add_subdirectory(src/string_algorithm)
add_subdirectory(src/advanced_algorithm)
add_subdirectory(src/bit_manipulation)
add_subdirectory(src/math)
add_subdirectory(tests)
```

`add_subdirectory` 将子目录纳入构建系统。每个子目录包含自己的 `CMakeLists.txt`，定义该模块的编译目标。这种分层组织方式使得：
- 每个数据结构/算法模块可独立定义源文件、头文件搜索路径和编译选项
- 新增模块只需添加一行 `add_subdirectory`，无需修改其他配置
- 模块间依赖关系清晰

#### 版本兼容性说明

| CMake 版本 | 发布日期 | 关键特性 | 本项目兼容性 |
|------------|----------|----------|-------------|
| 3.16 | 2019-11 | 预编译头文件、CUDA 语言改进 | 最低要求 |
| 3.20 | 2021-03 | C++23 标准预设、预设文件（presets） | 兼容 |
| 3.25 | 2022-11 | `CMAKE_COMPILE_WARNING_AS_ERROR` 变量 | 兼容 |

CMake 遵循向后兼容原则，使用 3.16 编写的 `CMakeLists.txt` 可在所有更高版本上正常工作。

#### 变更风险标注

⚠️ **极低风险** — CMake 的 `cmake_minimum_required` 机制通过策略（Policy）系统保证向后兼容。唯一需要关注的是：

- **CMake 4.0** 于 2025 年发布，移除了对 CMake 3.x 某些极旧策略的兼容支持。但由于本项目最低要求为 3.16，远高于被移除的旧策略版本，不受影响。
- 未来若需要使用 CMake Presets（`CMakePresets.json`），需将最低版本提升至 3.20+。目前本项目未使用 Presets，暂无此需求。

## 4 编译器

### 4.1 GCC 12+

#### 概述

GCC（GNU Compiler Collection）是 GNU 项目开发的编译器套件，是 Linux 生态中最广泛使用的编译器。GCC 支持 C、C++、Fortran、Ada、Go 等多种语言。本项目使用 GCC 作为 Linux 平台（CI 中的 `ubuntu-gcc` job）的主要编译器。

选择 GCC 12 作为最低版本的原因：

1. **C/C++17 完整支持**：GCC 12 对 C17 和 C++17 标准提供完整且经过充分验证的支持。
2. **改进的诊断信息**：GCC 12 相较于早期版本提供了更精确、更具可读性的警告和错误信息，包括更好的模板实例化错误提示。
3. **静态分析器改进**：GCC 10 引入的 `-fanalyzer` 静态分析选项在 GCC 12 中得到显著改进，可检测更多类型的缺陷。
4. ** sanitizer 支持**：GCC 12 对 AddressSanitizer、UndefinedBehaviorSanitizer 等运行时检测工具的支持更加成熟。

#### 官方文档

- [GCC 官方文档](https://gcc.gnu.org/onlinedocs/)
- [GCC 12 Release Notes](https://gcc.gnu.org/gcc-12/changes.html)
- [GCC C 标准支持状态](https://gcc.gnu.org/c99status.html)
- [GCC C++ 标准支持状态](https://gcc.gnu.org/projects/cxx-status.html)

#### 本项目使用方式

在 CI 配置 [.github/workflows/ci.yml](file:///d:/Dev/DevWorkSpace/VS%20Code/data-structures-and-algorithms/.github/workflows/ci.yml) 的 `ubuntu-gcc` job 中：

```yaml
- name: Install GCC-12
  run: |
    sudo apt-get update
    sudo apt-get install -y gcc-12 g++-12
    sudo update-alternatives --install /usr/bin/gcc gcc /usr/bin/gcc-12 100
    sudo update-alternatives --install /usr/bin/g++ g++ /usr/bin/g++-12 100
```

- `apt-get install -y gcc-12 g++-12`：从 Ubuntu 仓库安装 GCC 12 的 C 和 C++ 编译器。`-y` 标志自动确认安装，避免 CI 环境中的交互式提示。
- `update-alternatives --install`：将 `gcc-12` 和 `g++-12` 注册为系统默认的 `gcc` 和 `g++` 命令。优先级设为 100，确保覆盖 Ubuntu 镜像中预装的旧版本 GCC。这样后续的 `cmake` 配置步骤会自动检测到 GCC 12 作为默认编译器。

⚠️ **低风险** — GCC 保持严格的向后兼容。主要风险点：

- GCC 主版本升级可能引入新的警告，导致原本干净的构建输出出现新警告。建议定期在最新版 GCC 上验证构建。
- GCC 的 libstdc++ ABI 在 GCC 5 之后保持稳定（通过双 ABI 机制），不存在二进制兼容性风险。

## 5 代码质量工具

### 5.1 clang-format

#### 概述

clang-format 是 LLVM 项目提供的自动代码格式化工具，支持 C、C++、Objective-C、Java、JavaScript 等多种语言。它基于 LLVM 的词法分析器实现，能够精确解析代码结构，避免正则表达式方案常见的误格式化问题。

clang-format 的核心设计理念是"可配置的确定性格式化"——通过 `.clang-format` 配置文件定义格式规则，所有开发者执行格式化后得到完全一致的输出，从而消除代码评审中关于格式的争论。

#### 官方文档

- [clang-format 官方文档](https://clang.llvm.org/docs/ClangFormat.html)
- [clang-format 配置选项完整参考](https://clang.llvm.org/docs/ClangFormatStyleOptions.html)

#### 本项目使用方式

项目根目录的 [.clang-format](file:///d:/Dev/DevWorkSpace/VS%20Code/data-structures-and-algorithms/.clang-format) 配置逐项解析：

```yaml
BasedOnStyle: LLVM
```

**基于 LLVM 风格**：LLVM 风格是 clang-format 的内置风格之一，特点是缩进紧凑、括号同行（K&R 风格）。选择 LLVM 作为基础风格而非 Google 风格的原因：LLVM 风格的行宽默认为 80，括号风格更接近 C 语言传统，后续通过覆盖选项可灵活调整。Google 风格的行宽也为 80，但其对 `#include` 排序和命名有更严格的约定，不完全适合本项目。

```yaml
IndentWidth: 4
TabWidth: 4
UseTab: Never
```

- **缩进宽度 4**：4 空格缩进是 C/C++ 社区最广泛使用的缩进风格，在代码嵌套深度和可读性之间取得平衡。Linux 内核使用 8 空格，Google C++ 风格使用 2 空格，4 空格是折中选择。
- **Tab 宽度 4**：虽然本项目禁用 Tab（`UseTab: Never`），但设置 `TabWidth` 确保在遇到现有 Tab 字符时正确计算对齐。
- **禁用 Tab**：统一使用空格缩进，避免不同编辑器对 Tab 宽度解释不同导致的对齐混乱。这是现代 C/C++ 项目的普遍做法。

```yaml
BreakBeforeBraces: Allman
```

**Allman 括号风格**：也称为"BSD 风格"或"ANSI 风格"，要求左括号独占一行：

```c
if (condition)
{
    statement;
}
```

与之对比的是 K&R 风格（`BreakBeforeBraces: Attach`），左括号与控制语句同行：

```c
if (condition) {
    statement;
}
```

选择 Allman 风格的原因：
1. 括号独占一行使得代码块的开始和结束更加醒目，对初学者更友好。
2. 在数据结构与算法代码中，控制流嵌套较深时，Allman 风格更容易匹配括号对。
3. C 语言传统教学材料普遍使用 Allman 风格，与本项目教学定位一致。

```yaml
AllowShortFunctionsOnASingleLine: None
AllowShortIfStatementsOnASingleLine: Never
AllowShortLoopsOnASingleLine: false
```

- **禁止函数单行**：即使函数体只有一行，也必须换行书写。这确保所有函数具有一致的视觉结构，便于快速扫描代码。
- **禁止 if 单行**：禁止 `if (cond) stmt;` 形式，必须使用括号和换行。这避免了因后续添加语句时忘记添加括号而引入的缺陷（即"Apple SSL goto fail"类漏洞）。
- **禁止循环单行**：禁止 `while (cond) stmt;` 形式，理由同上。

```yaml
ColumnLimit: 100
```

**行宽限制 100**：每行代码的最大长度为 100 个字符。选择 100 而非传统的 80 的原因：
1. 现代显示器分辨率普遍为 1920×1080 或更高，100 字符行宽在分屏编辑时仍可完整显示。
2. C/C++ 代码中较长的标识符（如 `hash_table_open_addressing_insert`）和模板实例化容易超过 80 字符限制，100 字符可减少不必要的换行。
3. 100 字符是 LLVM 社区讨论中倾向采用的行宽，也是许多现代项目的选择。

```yaml
PointerAlignment: Left
```

**指针靠左对齐**：指针符号 `*` 靠近变量名而非类型名：

```c
int* ptr;    // PointerAlignment: Left
int *ptr;    // PointerAlignment: Right
int * ptr;   // PointerAlignment: Middle
```

选择靠左对齐的原因：`int* ptr` 更准确地表达"ptr 是一个 int 指针"的语义，且在多变量声明中不会产生误导（`int* a, b;` 中 `b` 是 `int` 而非 `int*`，靠左对齐反而提醒读者注意此陷阱）。

```yaml
SpaceBeforeParens: ControlStatements
```

**控制语句括号前加空格**：`if (`、`for (`、`while (` 等控制语句的括号前加空格，但函数调用 `func(` 不加空格。这是 C/C++ 社区最主流的风格约定，有助于在视觉上区分控制流和函数调用。

```yaml
SortIncludes: true
IncludeBlocks: Preserve
```

- **排序 include**：自动对 `#include` 指令按字母顺序排序，减少因排序不同导致的合并冲突。
- **保留 include 块**：不重新分组 `#include` 指令，仅在每个连续块内排序。开发者可通过空行分隔不同类别的头文件（如系统头文件、项目头文件），clang-format 会保留这些分组。

```yaml
AlignConsecutiveMacros: true
AlignConsecutiveDeclarations: false
```

- **对齐连续宏定义**：将连续的 `#define` 定义对齐，提升可读性：

```c
#define MAX_SIZE    100
#define MIN_SIZE    1
#define DEFAULT_SIZE 50
```

- **不对齐连续声明**：不对齐连续的变量声明。对齐声明虽然视觉整齐，但添加或修改变量名时会导致大范围无意义 diff，增加代码审查负担。

#### 版本兼容性说明

clang-format 随 LLVM 发布，版本与 LLVM 一致。`.clang-format` 配置文件中的选项在不同版本间基本向后兼容，但新版本可能引入新选项。

| LLVM 版本 | 发布日期 | 备注 |
|-----------|----------|------|
| 15 | 2022-09 | 本项目最低要求 |
| 17 | 2023-09 | 引入更多配置选项 |
| 20.1.3 | 2025-03 | 最新稳定版 |

#### 变更风险标注

⚠️ **极低风险** — clang-format 的格式化结果是确定性的，同一版本对同一输入始终产生相同输出。风险点：

- **跨版本格式化差异**：不同版本的 clang-format 对同一代码可能产生不同的格式化结果。建议团队统一使用同一版本，可通过 CI 中固定 LLVM 版本来保证。
- **`AlignConsecutiveMacros`** 在 LLVM 16 中从布尔值改为枚举类型（`ConsecutiveStyle`），但布尔值仍可向后兼容解析。

### 5.2 clang-tidy

#### 概述

clang-tidy 是 LLVM 项目提供的 C/C++ 代码静态分析工具（linter），基于 Clang 的语法树（AST）进行深度分析，能够检测出编译器警告无法覆盖的代码质量问题、潜在缺陷和现代化改进机会。与 clang-format 侧重格式不同，clang-tidy 侧重代码语义和逻辑质量。

clang-tidy 的检查分为多个模块，每个模块聚焦不同方面：`modernize-*` 推动代码现代化，`bugprone-*` 检测常见编程错误，`performance-*` 优化运行时性能，`readability-*` 提升代码可读性。

#### 官方文档

- [clang-tidy 官方文档](https://clang.llvm.org/extra/clang-tidy/)
- [clang-tidy 检查项完整列表](https://clang.llvm.org/extra/clang-tidy/checks/list.html)

#### 本项目使用方式

项目根目录的 [.clang-tidy](file:///d:/Dev/DevWorkSpace/VS%20Code/data-structures-and-algorithms/.clang-tidy) 配置逐项解析：

```yaml
Checks: >
  modernize-*,
  bugprone-*,
  performance-*,
  readability-*,
  -readability-magic-numbers,
  -readability-identifier-length,
  -readability-function-cognitive-complexity
```

**启用的检查模块**：

- **`modernize-*`**：检测可使用现代 C++ 特性替代的传统写法。例如：`auto` 推导替代显式类型声明、`nullptr` 替代 `NULL`、`override` 关键字标注虚函数重写、`using` 替代 `typedef`、范围 for 循环替代迭代器循环等。本项目使用 C++17，应充分利用现代特性。
- **`bugprone-*`**：检测常见编程错误模式。例如：sizeof 对指针类型误用、悬空引用、参数注释不匹配、sizeof 表达式中的整数运算错误、不正确的 `swap` 使用等。这是最有价值的检查模块之一，能捕获编译器无法发现的逻辑缺陷。
- **`performance-*`**：检测性能优化机会。例如：不必要的值拷贝（应传 `const` 引用）、不必要的 `shared_ptr` 拷贝、移动语义缺失等。数据结构与算法实现中性能至关重要，此模块帮助避免性能陷阱。
- **`readability-*`**：提升代码可读性。例如：`const` 正确性检查、括号一致性、命名规范、函数复杂度等。

**禁用的检查项**：

- **`-readability-magic-numbers`**：禁用对"魔术数字"（代码中直接出现的字面常量）的警告。数据结构与算法代码中大量使用有明确含义的数字（如数组大小、测试用例值），强制提取为命名常量反而降低代码可读性。
- **`-readability-identifier-length`**：禁用标识符长度检查。算法实现中广泛使用短变量名（如 `i`、`j`、`k` 作为循环变量，`n` 作为规模参数），这些是算法领域的通用约定，不应被标记为问题。
- **`-readability-function-cognitive-complexity`**：禁用函数认知复杂度检查。算法实现函数天然具有较高的认知复杂度（如动态规划的状态转移、回溯的递归逻辑），强制降低复杂度会导致代码结构碎片化，反而更难理解。

```yaml
HeaderFilterRegex: 'src/.*'
```

**头文件过滤正则**：仅对 `src/` 目录下的头文件进行检查。这避免了对外部库头文件或构建生成文件产生误报。正则表达式 `'src/.*'` 匹配所有以 `src/` 开头的头文件路径。

```yaml
CheckOptions:
  - key: modernize-use-override.AllowOverrideAndFinal
    value: true
  - key: modernize-use-nullptr.NullMacros
    value: 'NULL'
  - key: modernize-replace-auto-ptr.IncludeStyle
    value: llvm
  - key: bugprone-argument-comment.StrictMode
    value: false
  - key: bugprone-sizeof-expression.WarnOnSizeOfIntegerExpression
    value: true
  - key: performance-unnecessary-value-param.AllowedTypes
    value: ''
  - key: readability-braces-around-statements.ShortStatementLines
    value: 0
  - key: readability-const-return-type.IgnoreMacros
    value: true
  - key: readability-implicit-bool-conversion.AllowIntegerConditions
    value: false
  - key: readability-implicit-bool-conversion.AllowPointerConditions
    value: false
```

关键配置项解析：

- **`modernize-use-override.AllowOverrideAndFinal: true`**：允许同时使用 `override` 和 `final` 修饰符。某些场景下两者组合使用有明确语义（"此函数重写基类虚函数，且不可再被重写"）。
- **`modernize-use-nullptr.NullMacros: 'NULL'`**：将 `NULL` 宏视为应替换为 `nullptr` 的目标。C 代码中使用 `NULL`，C++ 代码中应使用 `nullptr`。
- **`bugprone-sizeof-expression.WarnOnSizeOfIntegerExpression: true`**：对 `sizeof(integer_expression)` 发出警告。这通常是编程错误——`sizeof` 应用于整数表达式返回的是整数类型的大小（通常为 4），而非开发者可能期望的数组大小。常见错误如 `sizeof(array)/sizeof(int)` 中误写为 `sizeof(array/sizeof(int))`。
- **`readability-braces-around-statements.ShortStatementLines: 0`**：要求所有控制语句（`if`、`for`、`while` 等）的执行体必须用括号包围，即使只有一行语句。值为 0 表示不允许任何单行无括号语句。
- **`readability-implicit-bool-conversion.AllowIntegerConditions: false`**：禁止在条件上下文中隐式转换整数为布尔值。即 `if (count)` 应写为 `if (count != 0)`，使意图更明确。
- **`readability-implicit-bool-conversion.AllowPointerConditions: false`**：禁止在条件上下文中隐式转换指针为布尔值。即 `if (ptr)` 应写为 `if (ptr != nullptr)`，使空指针检查意图更明确。

#### 版本兼容性说明

clang-tidy 随 LLVM 发布，版本与 clang-format 一致。检查项列表随版本增加，但已有检查项的行为保持稳定。

| LLVM 版本 | 检查项数量（约） | 新增关键检查 |
|-----------|----------------|-------------|
| 15 | ~500 | `modernize-use-ranges`、`bugprone-chained-comparison` |
| 17 | ~550 | `concurrency-*` 模块增强 |
| 20 | ~650+ | C++23 特性相关检查 |

#### 变更风险标注

⚠️ **低风险** — clang-tidy 的检查结果是建议性的，不影响编译。风险点：

- **新版本新增检查项**：升级 LLVM 版本后，`modernize-*` 等通配符会自动包含新增的检查项，可能产生新的警告。建议升级后审查新增检查项的适用性。
- **`CheckOptions` 键名变更**：LLVM 17 中部分 `CheckOptions` 的键名发生了重命名（如 `modernize-use-override.AllowOverrideAndFinal` 可能被移至其他命名空间），需关注 Release Notes。

## 6 文档生成

### 6.1 Doxygen

#### 概述

Doxygen 是最广泛使用的 C/C++ 文档生成工具，由 Dimitri van Heesch 开发。它从源代码中提取特殊格式的注释（Doc 注释），生成结构化的 HTML、LaTeX、PDF 等格式的文档。Doxygen 支持多种注释风格（Javadoc、Qt、XML 等），并能自动生成类层次图、协作图、调用图等可视化图表（需配合 Graphviz 的 `dot` 工具）。

#### 官方文档

- [Doxygen 官方文档](https://www.doxygen.nl/manual/index.html)
- [Doxygen 配置选项参考](https://www.doxygen.nl/manual/config.html)
- [Graphviz 官方网站](https://graphviz.org/)

#### 本项目使用方式

项目根目录的 [Doxyfile](file:///d:/Dev/DevWorkSpace/VS%20Code/data-structures-and-algorithms/Doxyfile) 关键配置解析：

```
PROJECT_NAME           = "Data Structures and Algorithms"
PROJECT_BRIEF          = "A comprehensive collection of data structures and algorithms implemented in C/C++"
```

- **项目名称**：显示在生成文档的标题栏和导航栏中。
- **项目简介**：显示在文档首页的项目描述，帮助读者快速了解项目定位。

```
OUTPUT_DIRECTORY       = doc/
```

**输出目录**：生成的文档文件存放于项目根目录下的 `doc/` 目录。此目录通常应加入 `.gitignore`，避免将生成的文档提交到版本控制。

```
INPUT                  = src/ tests/
INPUT_ENCODING         = UTF-8
FILE_PATTERNS          = *.c *.cpp *.h *.hpp
RECURSIVE              = YES
```

- **输入目录**：指定 `src/` 和 `tests/` 为文档提取的源文件目录。仅包含源代码和测试代码，不包含构建脚本和配置文件。
- **输入编码**：UTF-8 编码，确保中文注释正确解析。
- **文件模式**：仅处理 `.c`、`.cpp`、`.h`、`.hpp` 扩展名的文件，排除 `CMakeLists.txt`、Python 脚本等非 C/C++ 文件。
- **递归扫描**：递归遍历子目录，确保 `src/` 下所有层级的源文件都被处理。

```
EXTRACT_ALL            = YES
EXTRACT_PRIVATE        = NO
EXTRACT_STATIC         = YES
```

- **提取所有实体**：即使没有 Doc 注释，也为所有文档化实体（函数、类型、宏等）生成文档条目。这对教学项目很重要——即使开发者忘记添加注释，文档中仍会列出所有 API，只是描述为空。
- **不提取私有成员**：私有成员（C++ 中标记为 `private` 的成员）不生成文档。私有成员是实现细节，不属于公共 API。
- **提取静态成员**：静态函数和静态变量生成文档。C 语言中 `static` 函数是模块内部函数，但作为教学项目，了解内部实现同样重要。

```
GENERATE_HTML          = YES
GENERATE_LATEX         = NO
```

- **生成 HTML**：输出 HTML 格式文档，这是最常用的格式，可直接在浏览器中查看。
- **不生成 LaTeX**：跳过 LaTeX 输出。LaTeX 输出主要用于生成 PDF，但配置和编译 LaTeX 需要安装 TeX Live 等大型软件包，对大多数用户而言 HTML 输出已足够。

```
SOURCE_BROWSER         = YES
INLINE_SOURCES         = YES
```

- **源码浏览器**：在文档中提供源代码链接，点击函数名可跳转到对应的源代码位置。
- **内联源码**：在文档中直接嵌入函数体的源代码，无需跳转即可阅读实现。这对教学项目极为重要——学习者需要同时看到 API 文档和具体实现。

```
CALL_GRAPH             = YES
CALLER_GRAPH           = YES
HAVE_DOT               = YES
```

- **调用图**：为每个函数生成调用关系图，显示该函数调用了哪些其他函数。对理解算法执行流程非常有帮助。
- **调用者图**：为每个函数生成被调用关系图，显示哪些函数调用了该函数。有助于理解函数的使用场景。
- **启用 Graphviz**：使用 Graphviz 的 `dot` 工具生成调用图和类图。需系统安装 Graphviz（`sudo apt install graphviz` 或 `brew install graphviz`）。

```
WARN_IF_UNDOCUMENTED   = YES
WARN_IF_DOC_ERROR      = YES
WARN_NO_PARAMDOC       = YES
```

- **未文档化警告**：对没有 Doc 注释的实体发出警告，提醒开发者补充文档。
- **文档错误警告**：对 Doc 注释中的错误（如参数名不匹配、参数缺失等）发出警告。
- **缺少参数文档警告**：对函数声明中有但 Doc 注释中未描述的参数发出警告。

#### 版本兼容性说明

| Doxygen 版本 | 发布日期 | 关键特性 | 备注 |
|-------------|----------|----------|------|
| 1.9.x | 2021-2023 | 改进的 Markdown 支持、Emoji 支持 | 广泛使用 |
| 1.11.x | 2024 | 改进的 C++20 支持 | 稳定版 |
| 1.13.x | 2025 | 最新稳定版 | 本项目推荐版本 |

Doxygen 的配置文件格式在不同版本间保持高度兼容，`Doxyfile` 可直接升级使用。

#### 变更风险标注

⚠️ **极低风险** — Doxygen 是成熟的文档生成工具，配置格式稳定。风险点：

- **Graphviz 依赖**：调用图和类图功能依赖 Graphviz 的 `dot` 工具。若未安装 Graphviz，Doxygen 仍可正常生成文档，但不会包含图表。
- **HTML 输出样式**：不同版本的 Doxygen 生成的 HTML 样式可能有细微差异，但不影响内容。

## 7 持续集成

### 7.1 GitHub Actions

#### 概述

GitHub Actions 是 GitHub 提供的持续集成/持续部署（CI/CD）平台，于 2019 年正式发布。它允许开发者在 GitHub 仓库中定义自动化工作流（Workflow），响应代码推送、Pull Request 等事件自动执行构建、测试、部署等操作。

GitHub Actions 的核心概念：

- **Workflow（工作流）**：定义在 `.github/workflows/` 目录下的 YAML 文件，描述自动化流程。
- **Job（作业）**：工作流中的独立执行单元，运行在独立的虚拟机（Runner）上。同一工作流中的 Job 默认并行执行。
- **Step（步骤）**：Job 中的顺序执行步骤，可以是 Shell 命令或预定义的 Action。
- **Action（动作）**：可复用的操作单元，封装了常见任务（如检出代码、安装工具链等），通过 `uses` 关键字引用。

#### 官方文档

- [GitHub Actions 官方文档](https://docs.github.com/en/actions)
- [GitHub Actions 工作流语法](https://docs.github.com/en/actions/reference/workflow-syntax-for-github-actions)
- [GitHub Actions Runner 软件](https://github.com/actions/runner)

#### 本项目使用方式

项目的 CI 配置位于 [.github/workflows/ci.yml](file:///d:/Dev/DevWorkSpace/VS%20Code/data-structures-and-algorithms/.github/workflows/ci.yml)，包含三个并行 Job：

**触发条件**：

```yaml
on:
  push:
    branches: [master]
  pull_request:
    branches: [master]
```

- **push 到 master**：每次向 `master` 分支推送代码时触发 CI。
- **Pull Request 到 master**：每个针对 `master` 的 PR 都会触发 CI，确保合并前代码通过所有平台构建。

**Job 1：ubuntu-gcc**

```yaml
ubuntu-gcc:
  runs-on: ubuntu-latest
```

- 运行在 GitHub 提供的最新 Ubuntu 镜像上。`ubuntu-latest` 目前指向 Ubuntu 24.04 LTS。
- 安装 GCC-12 并通过 `update-alternatives` 设为默认编译器（详见 4.1 节）。
- 执行 CMake 配置（`cmake -B build`）和构建（`cmake --build build`）。
- 运行自测试（`cmake --build build --target test`），测试失败不阻塞 CI（`|| true`）。

**Job 2：macos-clang**

```yaml
macos-clang:
  runs-on: macos-latest
```

- 运行在 GitHub 提供的最新 macOS 镜像上。`macos-latest` 目前指向 macOS 14（Sonoma）。
- 使用 Xcode 自带的 Clang 编译器（详见 4.2 节）。
- 通过 Homebrew 安装 CMake。
- 构建和测试流程与 Ubuntu Job 相同。

**Job 3：windows-msvc**

```yaml
windows-msvc:
  runs-on: windows-latest
```

- 运行在 GitHub 提供的最新 Windows Server 镜像上。`windows-latest` 目前指向 Windows Server 2022。
- 使用 `ilammy/msvc-dev-cmd@v1` 配置 MSVC 开发环境（详见 4.3 节）。
- 通过 Chocolatey 安装 CMake，并添加到系统 PATH。
- 构建时指定 `--config Release`（MSVC 多配置生成器要求）。

**公共步骤**：

三个 Job 共享相同的核心步骤模式：

1. **Checkout**：`actions/checkout@v4` 检出仓库代码。`@v4` 是该 Action 的主版本，使用 Node.js 20 运行时，性能优于 v3。
2. **Install CMake**：各平台使用不同的包管理器安装 CMake（apt / brew / choco）。
3. **Configure**：`cmake -B build` 在 `build/` 目录中生成构建文件。`-B` 选项指定构建目录，支持外部构建（out-of-source build），保持源码目录整洁。
4. **Build**：`cmake --build build` 执行编译。CMake 作为构建系统生成器的统一前端，底层调用各平台的原生构建工具（Linux/macOS 为 Make，Windows 为 MSBuild）。
5. **Test**：`cmake --build build --target test` 执行 CTest 测试。`|| true` 确保测试失败不会导致整个 CI Job 失败——这在项目初期测试框架尚未完善时是合理的容错策略。

#### 版本兼容性说明

| 组件 | 当前版本 | 备注 |
|------|---------|------|
| `actions/checkout` | v4 | 基于 Node.js 20，推荐版本 |
| `ilammy/msvc-dev-cmd` | v1 | 稳定版本，支持 VS 2019/2022 |
| `ubuntu-latest` | Ubuntu 24.04 LTS | GitHub 定期更新 |
| `macos-latest` | macOS 14 Sonoma | GitHub 定期更新 |
| `windows-latest` | Windows Server 2022 | GitHub 定期更新 |

#### 变更风险标注

⚠️ **中等风险** — GitHub Actions 的运行环境由 GitHub 管理，存在以下风险：

- **Runner 镜像更新**：GitHub 会定期更新 `*-latest` 镜像中的预装软件版本，可能导致编译器版本隐式升级。例如 `ubuntu-latest` 预装的 GCC 版本可能从 12 升级到 13 或 14。
- **Action 版本**：`actions/checkout@v4` 等第三方 Action 可能发布新的主版本（v5），旧版本可能被弃用。建议定期检查并更新 Action 版本。
- **`|| true` 容错策略**：当前测试步骤使用 `|| true` 忽略失败，这在项目成熟后应移除，改为测试失败时阻断 CI，确保代码质量。

### 8.1 Python 3.10+

#### 概述

Python 在本项目中作为辅助工具语言，主要用于算法可视化、性能对比分析和数据验证。项目不使用 Python 实现核心数据结构或算法，而是利用 Python 丰富的科学计算和可视化生态来辅助 C/C++ 实现的开发与验证。

Python 3.10 于 2021 年 10 月发布，选择此版本作为最低要求的原因：

1. **结构模式匹配（Structural Pattern Matching）**：Python 3.10 引入了 `match-case` 语句，这是语言层面的重要新特性。
2. **类型提示增强**：Python 3.10 允许使用 `X | Y` 替代 `Union[X, Y]`，`list[X]` 替代 `List[X]`（从 `__future__` 导入 `annotations` 或直接使用），类型提示更简洁。
3. **性能改进**：Python 3.10 包含多项性能优化，对数据处理和可视化脚本有实际帮助。
4. **广泛可用性**：Python 3.10 已被所有主流操作系统和包管理器收录。

#### 本项目使用方式

项目中的 Python 脚本位于 `python/` 目录，包括：

- **`graph_visualization.py`**：图算法可视化脚本，将图的邻接矩阵/邻接表数据转换为可视化图形，帮助理解图的遍历、最短路径等算法的执行过程。
- **`hash_analysis.py`**：哈希表性能分析脚本，分析不同哈希函数的分布均匀性、冲突率和装载因子关系，为哈希表实现参数调优提供数据支撑。
- **`search_comparison.py`**：搜索算法性能对比脚本，对比线性搜索、二分搜索、插值搜索、斐波那契搜索等算法在不同数据规模和分布下的性能表现。
- **`sorting_comparison.py`**：排序算法性能对比脚本，对比各种排序算法的时间复杂度和实际运行时间，验证理论分析与实测结果的一致性。
