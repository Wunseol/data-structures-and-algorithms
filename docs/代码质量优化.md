### 2.2 文件组织规范

#### 当前问题

所有代码均在单一文件中，无头文件分离，导致：
- 类型定义与实现耦合
- 无法被其他模块复用
- 编译依赖过重

#### 目标结构

```
模块目录/
├── include/
│   └── module_name.h      # 类型定义、函数声明
├── src/
│   └── module_name.c      # 函数实现
├── test/
│   └── test_module.c      # 测试代码
└── main.c                  # 主程序入口（仅含 menu 和 main）
```

**头文件模板：**

```c
#ifndef MODULE_NAME_H
#define MODULE_NAME_H

#ifdef __cplusplus
extern "C" {
#endif

/* 类型定义 */
/* 函数声明 */

#ifdef __cplusplus
}
#endif

#endif /* MODULE_NAME_H */
```

### 2.4 错误处理规范

#### 当前问题

- `malloc` 失败后直接 `exit()`，无优雅退出
- `scanf` 返回值未检查，输入非法时行为未定义
- 数组越界无检查
- 函数声明有返回值但部分路径未返回

#### 目标规范

**C 语言错误处理模式：**

```c
typedef enum {
    DS_OK = 0,
    DS_ERROR = -1,
    DS_OVERFLOW = -2,
    DS_UNDERFLOW = -3,
    DS_NOT_FOUND = -4,
    DS_INVALID_PARAM = -5,
    DS_NULL_PTR = -6
} DSStatus;
```

**输入验证模式：**

```c
if (scanf("%d", &n) != 1) {
    fprintf(stderr, "输入格式错误\n");
    while (getchar() != '\n');
    continue;
}
if (n < 0 || n > MAX_SIZE) {
    fprintf(stderr, "输入范围错误: %d (有效范围: 0-%d)\n", n, MAX_SIZE);
    continue;

```

    #include <conio.h>

#else
    #include <termios.h>
    #include <unistd.h>

        int ch;

        newt = oldt;

        return ch;
    }
    #define GET_CHAR() getch()

| — | C++26 | 制定中 | 🔮 草案 | `std::linalg`（线性代数）、反射、契约、`std::execution`（执行器） | 尚无完整支持 |

**时效性注意事项：**
- C23 标准已于 2024 年正式发布，但截至 2026 年 5 月，主流编译器仍在逐步实现中。项目应优先遵循 C17，在确认编译器支持的前提下可选用 C23 特性
- C++23 标准已于 2023 年发布，核心特性（`std::expected`、`std::print`）在 GCC 14+ 和 MSVC 19.35+ 中已有初步支持
- C++26 仍在制定中，`std::linalg` 线性代数库与本项目矩阵运算需求相关，值得持续关注

### 6.4 静态分析工具

| 工具 | 版本 | 功能 | 安装方式 | 备注 |
|------|------|------|---------|------|
| clang-tidy | LLVM 17+ | C/C++ 代码风格和 bug 检查 | 随 LLVM/Clang 安装 | 支持 `.clang-tidy` 配置文件，可自定义规则集 |
| cppcheck | 2.14+ | C/C++ 静态分析，专注 bug 检测 | `apt install cppcheck` / `choco install cppcheck` | 误报率较低，适合 CI 集成 |
| AddressSanitizer (ASan) | GCC/Clang 内置 | 运行时内存错误检测 | 编译选项 `-fsanitize=address` | 检测：缓冲区溢出、use-after-free、内存泄漏 |
| UndefinedBehaviorSanitizer (UBSan) | GCC/Clang 内置 | 运行时未定义行为检测 | 编译选项 `-fsanitize=undefined` | 检测：整数溢出、空指针解引用、类型不匹配 |
| MemorySanitizer (MSan) | Clang 内置 | 运行时未初始化内存读取检测 | 编译选项 `-fsanitize=memory` | 仅 Clang 支持，与 ASan 互斥 |
| Valgrind | 3.22+ | 运行时内存错误和性能分析 | `apt install valgrind` | Linux 专用，功能最全面但速度较慢 |
| CppDepend | 2024.1 | 代码度量、依赖分析、架构规则 | 商业工具 | 适合大型项目架构治理 |

**推荐工具组合：**
- 开发阶段：`clang-tidy`（编辑器集成）+ `cppcheck`（提交前检查）
- 测试阶段：`ASan` + `UBSan`（编译选项，运行测试）
- Linux 环境：`Valgrind`（深度内存分析）

### 6.5 编译器警告选项

**GCC/Clang 推荐警告选项：**

```bash
# 基础警告（必须零警告）
-Wall -Wextra -Wpedantic

# 额外有用警告
-Wconversion          # 隐式类型转换
-Wshadow              # 变量遮蔽
-Wformat=2            # printf/scanf 格式字符串检查
-Wnull-dereference    # 空指针解引用
-Wdouble-promotion    # float 隐式提升为 double
-Werror               # 将警告视为错误（CI 环境推荐）
```

**MSVC 推荐警告选项：**

```bash
/W4     # 高级别警告
/permissive-  # 严格标准一致性
/sdl    # 安全开发生命周期检查
```

## 7. 验收标准

### 7.1 逻辑 Bug 修复验收

| 指标 | 标准 | 验证方式 |
|------|------|---------|
| 严重 bug 清零 | 第 3 节中所有 🔴 标记的问题均已修复 | 逐项核对 |
| 回归测试 | 修复后原有正确功能不受影响 | 功能测试 |
| 边界用例 | 每个 bug 修复至少 3 个测试用例（正常、边界、异常） | 测试记录 |

### 7.2 内存安全验收

| 指标 | 标准 | 验证方式 |
|------|------|---------|
| AddressSanitizer | 所有模块 ASan 检测零错误 | ASan 运行报告 |
| 内存泄漏 | 所有 `malloc/new` 有对应 `free/delete` | 代码审查 + ASan |
| 悬空指针 | 无 use-after-free | ASan 运行报告 |

### 7.3 编码规范验收

| 指标 | 标准 | 验证方式 |
|------|------|---------|
| 编译警告 | `gcc -Wall -Wextra -Wpedantic` 零警告 | 编译输出 |
| clang-tidy | 无严重级别警告 | clang-tidy 报告 |
| 命名规范 | 函数名与功能一致，无误导命名 | 代码审查 |
| C/C++ 混用 | 同一文件内不混用 `malloc/free` 与 `new/delete` | 代码审查 |

### 7.4 模块化验收

| 指标 | 标准 | 验证方式 |
|------|------|---------|
| 头文件分离 | 每个模块有独立的 `.h` 和 `.c/.cpp` 文件 | 文件结构检查 |
| 全局变量消除 | 无不必要的全局变量 | 代码审查 |
| 接口一致性 | 同类数据结构提供统一接口 | 接口文档 |

### 7.5 跨平台验收

| 指标 | 标准 | 验证方式 |
|------|------|---------|
| 平台依赖 | 无直接调用 `system("cls")`、`getch()`、`system("pause")` | 代码搜索 |
| 条件编译 | 平台相关代码使用 `#ifdef` 封装 | 代码审查 |
| 多平台编译 | 至少在 Windows (MSVC/GCC) 和 Linux (GCC) 上编译通过 | CI 构建 |
